# 보건온 디자인 시스템

## 1. 문서 목적

이 문서는 보건온의 시각 언어, 디자인 토큰, 컴포넌트 표현, 반응형 기준과 차별성 검증 규칙을 정의한다.

보건온은 `Daily Briefing`, `Calm Professional`, `Personal Operations`를 핵심 콘셉트로 사용한다. 귀엽고 말랑한 교사용 앱보다 차분하고 신뢰할 수 있는 개인 업무 운영 도구에 가까운 인상을 만든다.

## 2. 디자인 원칙

### 2.1 운영 브리핑 우선

오늘의 행동, 상태와 다음 일정, 월간 흐름을 한 화면에서 함께 읽을 수 있어야 한다. 캘린더는 중앙 조망 영역이고 오른쪽 운영 열은 즉시 행동 영역이다.

### 2.2 차분한 전문성

- 깊은 네이비를 브랜드 기반으로 사용한다.
- 청록은 핵심 행동과 선택 상태에 제한한다.
- 장식보다 정보 위계를 우선한다.
- 과도한 그라데이션, 유리 효과와 강한 그림자를 사용하지 않는다.

### 2.3 업무 밀도

Notion 수준의 정보 밀도와 Linear처럼 명확한 상태 표현을 지향한다.

- 모든 정보를 카드로 감싸지 않는다.
- 목록과 구분선을 적극 사용한다.
- 핵심 영역에만 표면과 배경 차이를 준다.
- 넓은 빈 공간보다 읽기 쉬운 간격을 사용한다.

### 2.4 독립적 시각 정체성

일반적인 내비게이션, 카드, 패널과 캘린더 패턴은 사용할 수 있다. 그러나 특정 서비스와 같은 배치·비율·색상·메뉴 순서·오늘 표시·빈 상태 조합을 재현하지 않는다.

## 3. 금지되는 시각 조합

- 민트를 메인 컬러로 채운 거의 흰색 대시보드
- 특정 레퍼런스의 사이드바·달력·오른쪽 카드 비율과 장식을 그대로 복제한 조합
- 상단에 날짜·시간·교시와 큰 계정 연동 버튼을 나란히 둔 구성
- 모바일 하단 메뉴 위 대형 민트 원형 추가 버튼
- 크고 둥근 흰색 카드를 반복하는 화면
- 넓고 높은 빈 달력 셀
- 오늘 날짜를 민트색 채움 원형으로 표현
- 특정 서비스와 유사한 월·연도 선택기, 달력 외곽선과 빈 상태 문구

- 비활성 상태도 읽을 수 있어야 하며 투명도만 과도하게 낮추지 않는다.

## 5. 타이포그래피

### 5.1 글꼴

권장 순서:

```css
font-family: Pretendard, "Noto Sans KR", system-ui, -apple-system,
  BlinkMacSystemFont, "Segoe UI", sans-serif;
```

제품은 `@fontsource/pretendard`를 통해 400·500·600·700 굵기를 자체 호스팅한다. 숫자와 시간은 동일 글꼴의 tabular 숫자 기능을 사용한다.

### 5.2 크기 토큰

| 토큰 | 크기/행간 | 굵기 | 용도 |
|---|---|---|---|
| `text-3xl` | 26/35 | 700 | 브리핑 날짜·핵심 수치 |
| `text-2xl` | 22/30 | 700 | 페이지 제목·수치 |
| `text-xl` | 18/24 | 700 | 주요 섹션·워드마크 |
| `text-lg` | 16/22 | 700 | 카드·패널 제목 |
| `text-base` | 14/21 | 400~600 | 기본 본문·목록 제목 |
| `text-md` | 13/20 | 400~600 | 입력·목록 메타 정보 |
| `text-sm` | 12/18 | 500~600 | 배지·날짜·보조 라벨 |
| `text-xs` | 11/16 | 500~600 | 캘린더·제한된 보조 정보 |
| `text-2xs` | 10/15 | 600 | 필터의 짧은 상단 라벨 |

### 5.3 타이포그래피 규칙

- 모바일에서도 기본 본문을 14px보다 작게 축소하지 않는다.
- 긴 업무 제목은 최대 두 줄, 달력 셀 제목은 한 줄로 제한한다.
- 시간, 수량과 진행률에는 tabular 숫자를 적용한다.
- 영문 대문자 추적을 장식 요소로 남용하지 않는다.

## 6. 간격 시스템

2px 보조 단위를 포함한 4px 기반 밀도 토큰을 사용한다.

| 토큰 | 값 |
|---|---:|
| `space-1` | 2px |
| `space-2` | 4px |
| `space-3` | 6px |
| `space-4` | 8px |
| `space-5` | 10px |
| `space-6` | 12px |
| `space-7` | 14px |
| `space-8` | 16px |
| `space-10` | 20px |
| `space-12` | 24px |
| `space-14` | 28px |
| `space-16` | 32px |
| `space-20` | 40px |

규칙:

- 목록 행 내부: 12~16px
- 관련 필드 사이: 12px
- 섹션 내부: 16~24px
- 주요 섹션 사이: 28~40px
- PC 캔버스 좌우: 24~40px
- 모바일 화면 좌우: 16px
- 넓은 빈 공간으로 정보량을 의도치 않게 줄이지 않는다.

## 7. 반경과 그림자

### 7.1 반경

| 토큰 | 값 | 용도 |
|---|---:|---|
| `radius-sm` | 6px | 배지·작은 입력 |
| `radius-md` | 10px | 버튼·입력·목록 선택 |
| `radius-lg` | 14px | 핵심 카드·패널 |
| `radius-xl` | 18px | 모달·모바일 시트 |
| `radius-full` | 999px | 작은 점·상태 필 |

모든 컨테이너에 큰 반경을 적용하지 않는다. 목록, 캘린더와 설정 구역은 선과 표면 차이만으로 구분할 수 있다.

### 7.2 그림자

```css
--shadow-sm: 0 1px 2px rgb(16 31 51 / 0.06);
--shadow-md: 0 8px 24px rgb(16 31 51 / 0.10);
--shadow-panel: -12px 0 32px rgb(16 31 51 / 0.12);
```

- 핵심 카드에는 `shadow-sm`만 사용하고 표 형태 목록은 선과 표면 차이로 구분한다.
- 떠 있는 메뉴, 상세 패널, 모달과 시트에만 제한한다.
- 여러 겹의 밝은 외곽 그림자를 사용하지 않는다.

## 8. 레이아웃

### 8.1 중단점

| 이름 | 범위 | 기본 동작 |
|---|---|---|
| 모바일 | 0~767px | 하단 메뉴, 단일 열 |
| 태블릿 | 768~1023px | 축약 사이드바 또는 2열 운영 화면 |
| 데스크톱 | 1024~1439px | 고정 사이드바, 중앙·오른쪽 운영 열 |
| 와이드 | 1440px 이상 | 3열 운영 대시보드, 상세 패널 오버레이 |

### 8.2 최대 너비

- 기본 작업 캔버스: 1440px
- 텍스트 중심 설정 화면: 880px
- 로그인 콘텐츠: 440px
- 우측 상세 패널: 420~480px

### 8.3 PC 브리핑 그리드

```css
grid-template-columns: minmax(0, 2fr) minmax(300px, 1fr);
gap: 20px;
```

App Shell의 220px 사이드바 옆에서 중앙 캘린더와 오른쪽 운영 열을 구성한다. 태블릿에서는 중앙과 오른쪽 모듈을 2열 또는 순차 카드로 재배치하고, 모바일에서는 오른쪽 운영 열을 본문 아래로 이동한다.

### 8.4 데스크톱 사이드바

- 너비 220px, 화면 높이에 고정
- 네이비 워드마크와 `새로 만들기` 메뉴를 위쪽에 배치
- 메뉴는 `보건업무`, `나의 기록`, `설정`으로 그룹화한다.
- 사용자 화면에서는 Workflow를 `업무 절차`로 표기한다.
- `새로 만들기` 메뉴는 업무, 일정, 업무 절차, 빠른 입력, 작성 도움 진입점을 제공한다.
- 작성 도움은 독립적인 주 메뉴가 아니라 새로 만들기 또는 상세 화면에서만 노출한다.
- 동기화 상태와 프로필은 아래쪽에 배치
- 활성 메뉴는 옅은 표면과 2px 인디케이터로 표시

### 8.5 운영 대시보드

- 전체: 220px 사이드바 + 중앙 `minmax(0, 2fr)` + 오른쪽 `minmax(300px, 1fr)`
- 중앙: 날짜·근무 상태, 주간 요약, 월간 통합 캘린더
- 오른쪽: 오늘 업무, 마감 임박, 진행 중 업무 절차, 오늘 일정과 빠른 메모 진입
- 오른쪽 모듈은 개별 대형 카드 대신 하나의 표면 안에서 구분선으로 묶는다.
- 편집 상세 패널은 운영 열과 별도로 오버레이된다.

## 9. 표면과 섹션

### 9.1 기본 캔버스

페이지 배경은 아이보리 또는 쿨 그레이다. 모든 내부 영역을 흰색 카드로 만들지 않는다.

### 9.2 표면 사용

화이트 표면을 사용하는 곳:

- 로그인 영역
- 축소 달력
- 핵심 오늘 꼭 끝낼 일 영역
- 상세 패널
- 모달·시트
- 브리핑 요약과 운영 모듈
- 업무·템플릿·업무 절차 카드

표면 없이 사용하는 곳:

- 브리핑의 목록 섹션
- 시간순 일정
- 설정 섹션
- 프로젝트 목록
- 선택 날짜 간단 목록

### 9.3 섹션 구분

- 섹션 제목
- 1px 구분선
- 간격
- 필요할 때만 옅은 배경

큰 둥근 카드 반복보다 위 요소를 우선한다.

## 10. 버튼

### 10.1 주요 버튼

- 배경: `action-700`
- 텍스트: 흰색
- 높이: PC 40px, 모바일 44px
- 반경: `radius-md`
- 아이콘은 왼쪽 16px 크기

예: `+ 새로 만들기`, `저장`

### 10.2 보조 버튼

- 흰색 또는 투명 배경
- `line-strong` 테두리 또는 테두리 없음
- 네이비 텍스트

### 10.3 텍스트 버튼

- 배경 없음
- 청록 또는 네이비 텍스트
- `전체 캘린더 열기`, `전체 보기`에 사용

### 10.4 위험 버튼

- 코랄·레드 계열
- 삭제 확인 화면에서만 사용
- 주요 저장 버튼과 인접 배치 시 충분한 간격 확보

### 10.5 금지

- 하단 메뉴 위 대형 원형 추가 버튼
- 모든 버튼을 청록색으로 표시
- 계정 상태를 큰 테두리형 연동 버튼으로 상시 표시

## 11. 입력 요소

- PC 기본 높이: 40px
- 모바일 기본 높이: 44px
- 테두리: `line-strong`
- 포커스: 2px `action-500` 외곽선
- 오류: `error` 테두리와 필드 아래 메시지
- 라벨은 placeholder로 대체하지 않는다.
- 긴 폼은 기본 정보와 세부 입력을 구분한다.
- 학생 개인정보 금지 안내는 메모·체크리스트 가까이에 둔다.

## 12. 카드

### 12.1 핵심 카드

사용 가능:

- 오늘 꼭 끝낼 일
- 축소 달력
- 오늘 운동 한 건
- 로그인 영역

스타일:

- `radius-lg`
- 1px `line`
- 그림자 없음 또는 `shadow-sm`
- 패딩 20~24px

### 12.2 데일리 대시보드 카드

- 반경은 14~18px 범위에서 외부 컨테이너와 내부 요소를 구분한다.
- 카드 간격은 16~20px을 사용한다.
- 제목, 설명, 메타 정보, 동작의 순서가 항상 읽히도록 구성한다.
- 높이는 내용에 맞추며 카드끼리 억지로 같은 높이를 만들지 않는다.
- 상태색은 배지와 작은 아이콘에만 사용하고 카드 전체를 채우지 않는다.
- 그림자는 `shadow-sm` 이하로 제한한다.

### 12.3 카드 금지 패턴

- 모든 섹션을 같은 크기 카드로 반복
- 과도하게 큰 제목과 넓은 내부 여백
- 카드마다 다른 파스텔 배경
- 카드가 정보보다 장식적으로 더 강한 구조

## 13. 목록과 업무 행

### 13.1 기본 행

- 최소 높이: 52px
- 상하 패딩: 10~12px
- 행 사이 1px 구분선
- 제목과 메타 정보의 2단 정보 구조
- hover는 `brand-50`
- 선택은 `action-50`과 왼쪽 3px 청록선

### 13.2 정보 순서

1. 완료 또는 유형 아이콘
2. 제목
3. 영역·태그
4. 상태·우선순위
5. 날짜
6. 행 동작

모바일은 2~4를 두 번째 줄에 합치고 날짜는 오른쪽 정렬한다.

### 13.3 업무 카드 동작

- 기본 노출은 세부 항목, 편집, 더보기로 제한한다.
- 복제, 템플릿 저장, 내용 제안, 삭제는 더보기 메뉴에 둔다.
- 동작 영역은 제목과 메타 정보를 압박하거나 한국어 라벨을 어색하게 줄바꿈하지 않는다.
- 템플릿 카드는 데스크톱 3열, 태블릿 2열, 모바일 1열을 사용하며 적용 버튼과 더보기 메뉴를 분리한다.

### 13.4 빈 상태

- 비어 있는 주요 화면은 중앙 정렬된 아이콘, 제목, 설명, 주요 동작을 제공한다.
- 빈 상태는 상단의 짧은 문구나 구분선만으로 끝내지 않는다.
- 주요 동작은 기존 생성 패널을 재사용하고 별도 데이터 경로를 만들지 않는다.

## 14. 배지

- 높이: 22~24px
- 반경: `radius-sm` 또는 `radius-full`
- 글자: `label-sm`
- 좌우 패딩: 7~9px
- 상태명 또는 영역명을 반드시 포함

상태별 아이콘 예:

- 예정: 원형 윤곽
- 진행 중: 반원 진행 아이콘
- 회신 대기: 말풍선 또는 시계
- 확인 필요: 체크가 없는 돋보기
- 완료: 체크
- 보류: 일시정지

아이콘만으로 상태를 전달하지 않는다.

## 15. 동기화 상태

상단 우측 프로필 옆에 작고 조용하게 표시한다.

| 상태 | 표현 |
|---|---|
| 동기화됨 | 작은 성공 점 + `동기화됨` |
| 저장 중 | 회전 아이콘 + `저장 중` |
| 방금 저장됨 | 체크 + `방금 저장됨` |
| 저장 실패 | 코랄 경고 + `저장 실패` |
| 연결 끊김 | 슬래시 클라우드 + `연결 끊김` |

- 높이 28px 이하
- 큰 배너나 버튼으로 만들지 않는다.
- 모바일에서는 아이콘을 기본 표시하고 접근 가능한 이름을 제공한다.
- 오류와 연결 끊김은 필요할 때 짧은 텍스트를 함께 표시한다.

## 16. 프로필 아바타

- PC: 32px
- 모바일: 30px
- 원형 이미지 또는 이니셜
- 얇은 중성 테두리
- 선택 시 계정 메뉴 열기
- 큰 Google 계정 연동 문구를 옆에 상시 표시하지 않는다.

## 17. 우측 상세 패널

- 너비: 420~480px
- 배경: `surface`
- 왼쪽 구분선과 `shadow-panel`
- 상단: 제목, 상태, 닫기
- 본문: 스크롤
- 하단: 저장·취소 또는 편집·삭제
- 반경은 화면 내부 도킹 시 사용하지 않음
- 오버레이형일 때 왼쪽 모서리에만 `radius-xl` 사용 가능

## 18. 모바일 하단 시트

- 최대 높이: 화면의 92%
- 상단 모서리: `radius-xl`
- 드래그 핸들: 중성 회색 36×4px
- 하단 안전 영역 포함
- 저장 버튼은 전체 폭
- 배경 딤은 네이비 24% 투명도

새 항목 생성 시에만 사용하며 대형 원형 버튼과 결합하지 않는다.

## 19. 캘린더 공통 토큰

| 토큰 | 값 |
|---|---:|
| 요일 행 높이 | 28px |
| 축소 셀 최소 높이 | 46px |
| PC 전체 월간 셀 최소 높이 | 92px |
| 모바일 월간 셀 높이 | 48~56px |
| 셀 내부 패딩 | 5~8px |
| 그리드 선 | `line-subtle` |
| 선택 선 | 1.5px `action-600` |

셀 높이는 제목 수에 따라 늘어나지 않는다.

## 20. 축소 월간 달력

### 20.1 구조

- 조밀한 7열 그리드
- 전체 너비 320~420px 범위
- 날짜 숫자 중심
- 한 셀에 최대 2개의 짧은 마커

### 20.2 오늘

- 날짜 숫자 오른쪽 위 4px 네이비 점 또는 숫자 아래 2px 네이비 선
- 셀 전체 채움 금지
- 원형 민트 채움 금지

### 20.3 선택 날짜

- `action-50` 배경
- `action-600` 얇은 테두리
- 날짜 숫자는 `brand-900`

### 20.4 날짜 상태 마커

- 일정: 영역색 4px 점
- 마감: 코랄 4px 짧은 선 또는 작은 깃발 아이콘
- 운동: 코발트 링
- 초과 항목: `+N` caption

색상만으로 구분하지 않고 마감과 운동에는 형태를 다르게 사용한다.

## 21. 전체 월간 캘린더

- 화면 너비를 활용하지만 달력 위아래의 빈 공간을 과도하게 만들지 않는다.
- 셀 안 제목은 한 줄, 최대 2개다.
- 세 번째부터 `+N개`로 표시한다.
- 기간 일정은 얇은 영역색 바와 한 줄 제목을 사용한다.
- 수행일 할 일은 체크박스형 아이콘을 사용한다.
- 마감일은 깃발 아이콘을 사용한다.
- 운동은 운동 유형 아이콘 또는 코발트 링을 사용한다.

## 22. 주간 캘린더

- 종일·업무 영역 높이: 내용에 따라 최대 96px 후 스크롤 또는 `+N`
- 시간 축 간격: 60분 기본, 시각 라벨은 왼쪽
- 현재 시간: 코랄 또는 네이비 1px 선과 작은 `현재` 라벨
- 일정 블록: 영역색의 옅은 배경과 진한 왼쪽선
- 둥근 알약 모양을 반복하지 않는다.
- 드래그 핸들과 리사이즈 커서를 표시하지 않는다.

## 23. 모바일 주간 스트립

- 7열 동일 너비
- 높이: 64~72px
- 요일: label-sm
- 날짜: body-md 또는 heading-md
- 선택: 하단 청록선 또는 옅은 직사각 배경
- 오늘: 작은 네이비 점
- 일정 존재: 날짜 아래 작은 점 최대 2개

각 날짜를 큰 원형 버튼으로 만들지 않는다.

## 24. 모바일 월간 달력

- 화면 좌우 16px 안에서 7열 유지
- 셀 높이 48~56px 고정
- 긴 제목 표시 금지
- 영역 점 최대 3개
- 마감은 작은 깃발 아이콘
- 초과 시 작은 수량 표시
- 선택 날짜 상세 목록을 달력 아래에 배치

## 25. 타임라인

- 왼쪽 시간 열: 64px
- 오른쪽 내용 열: 나머지 너비
- 시간 사이 얇은 구분선
- 현재 시각 선은 전체 너비를 가로지르지 않고 일정 영역 안에서만 표시
- 일정 제목, 영역과 장소를 최대 2줄 안에 배치

## 26. 아이콘

Lucide Icons를 기본으로 사용한다.

- 기본 크기: 16px
- 주요 버튼: 18px
- 모바일 하단 메뉴: 22px
- stroke width: 1.75~2
- 같은 기능에 같은 아이콘 사용
- 수달 캐릭터를 메뉴 아이콘으로 사용하지 않음

## 27. 수달 캐릭터

허용:

- 빈 상태
- 온보딩
- 도움말
- 완료 상태

규칙:

- 데스크톱 최대 120px, 모바일 최대 96px
- 핵심 정보나 버튼보다 먼저 시선을 끌지 않음
- 반복 애니메이션 금지
- 대시보드 중심 일러스트 금지
- 주요 메뉴 아이콘 금지

## 28. 모션

| 동작 | 시간 | easing |
|---|---:|---|
| hover·색 변화 | 120ms | ease-out |
| 패널 열기 | 200ms | cubic-bezier(0.2, 0.8, 0.2, 1) |
| 하단 시트 | 220ms | cubic-bezier(0.2, 0.8, 0.2, 1) |
| 토스트 | 160ms | ease-out |

- 장식용 부유 애니메이션을 사용하지 않는다.
- `prefers-reduced-motion`에서 이동 모션을 제거하거나 즉시 전환한다.

## 29. 로딩과 스켈레톤

- 실제 화면 구조와 같은 행·텍스트 형태 사용
- 둥근 대형 카드 스켈레톤 반복 금지
- 애니메이션 대비를 낮게 유지
- 기존 데이터가 있으면 전부 스켈레톤으로 교체하지 않고 갱신 상태만 표시

## 30. 빈 상태

- 작은 수달 또는 단순 아이콘
- 독립적인 한 문장
- 한 개의 주요 행동
- 배경 전체를 민트로 채우지 않음
- 큰 흰색 카드 중앙에 과도한 여백으로 배치하지 않음

예시 문구:

- `오늘 지정한 핵심 업무가 없습니다.`
- `선택한 날짜에 등록된 항목이 없습니다.`
- `기다리는 회신이 없습니다.`
- `첫 운동 일정을 기록해 보세요.`

## 31. 오류와 피드백

- 인라인 오류: 필드 아래
- 페이지 조회 오류: 해당 섹션 안
- 전역 연결 오류: 상단 얇은 상태 바
- 저장 성공: 짧은 토스트와 동기화 상태
- 위험 확인: 모달
- 오류에 코랄을 사용하되 화면 전체를 붉게 채우지 않는다.

## 32. 접근성

- 텍스트 대비 WCAG AA 목표
- 포커스 링 2px `action-500`, 바깥 여백 2px
- 터치 대상 최소 44×44px
- 아이콘 단독 버튼에 접근 가능한 이름
- 상태와 영역을 색상만으로 표현하지 않음
- 달력 셀에 날짜, 일정 수, 마감 여부를 접근 가능한 이름으로 제공
- 키보드 화살표로 달력 날짜 이동
- 모달·패널·시트 포커스 관리
- 동기화 상태를 스크린 리더에 과도하게 반복 공지하지 않음

## 33. 반응형 전환

### PC → 태블릿

- 사이드바를 아이콘 중심의 축약형으로 전환
- 중앙·오른쪽 2열을 유지할 공간이 없으면 오른쪽 운영 열을 중앙 아래로 이동
- 우측 상세 패널을 오버레이로 전환

### 태블릿 → 모바일

- 사이드바를 하단 5개 메뉴로 전환
- 축소 월간 달력을 7일 주간 스트립으로 교체
- 새로 만들기는 상단 텍스트·아이콘 버튼
- 상세 패널을 전체 화면 상세 또는 하단 시트로 전환

화면 폭 변경으로 정보 의미나 메뉴 용어를 변경하지 않는다.

- [ ] 카드 크기와 배치가 유사하지 않은가?
- [ ] 달력 비율과 오늘 표시가 독립적인가?
- [ ] 메뉴명과 빈 상태 문구가 독립적인가?
- [ ] 브리핑 중심 흐름이 분명한가?
- [ ] 달력이 일정관리 기능으로 충분히 제공되는가?

## 37. Phase 5 컴포넌트 규칙

- 템플릿·연간 월·체크리스트는 기존 `surface`, `line`, `shadow-sm`, `radius-md` 토큰을 사용한다.
- 템플릿 카드 정보는 이름, 카테고리, 설명, 예상 시간, 권장 시기, 체크리스트 수 순으로 읽힌다.
- 체크리스트와 링크 행은 조밀한 `control-sm` 높이를 사용하되 모바일 삭제·이동 버튼의 터치 영역은 식별 가능하게 유지한다.
- 연간 월 카드의 상태는 색상만으로 표현하지 않고 `반복`, `완료`, `일정` 텍스트를 함께 표시한다.
- 빠른 입력 미리보기는 민트 계열 보조 배경을 사용하되 저장 버튼과 혼동되지 않도록 `폼에 반영` 행동을 분리한다.
- 새 화면은 Pretendard 타이포그래피, 기존 사이드바와 모바일 하단 내비게이션을 그대로 사용한다.

## 38. Phase 6 Workflow OS 컴포넌트 규칙

### 38.1 상태 표현

- Workflow 상태는 영문 저장값 대신 `진행 중`, `일시 중지`, `완료`, `취소` 레이블을 표시한다.
- 단계 상태는 `대기`, `진행 중`, `완료`, `건너뜀`, `보류` 레이블을 사용한다.
- 상태는 색상만으로 구분하지 않고 레이블과 아이콘을 함께 사용한다. 진행 중은 브랜드 민트, 일시 중지·보류는 주의색, 완료는 성공색, 취소·건너뜀은 중성색을 사용한다.
- 완료·취소 Workflow는 종료 상태임을 명시하고 활성 상태처럼 보이는 강조 버튼을 표시하지 않는다.

### 38.2 진행률과 다음 행동

- 진행률 컴포넌트는 막대와 `완료 단계/전체 단계`, 백분율을 함께 표시한다. 건너뛴 단계는 완료율에 포함하지 않되 남은 단계에서도 제외한다.
- `다음 행동`은 한 카드에 한 행동만 강조하고 단계명과 체크리스트 잔여 수를 보조 정보로 표시한다.
- 진행률 막대의 최소 높이는 8px이며 대비를 유지한다. 숫자 정보는 스크린 리더가 한 문장으로 읽을 수 있는 레이블을 제공한다.

### 38.3 단계 타임라인

- Desktop 단계 행은 상태 아이콘, 순번·이름, 상태·예상 시간, 설명, 전이 행동, 확장 편집 순으로 읽힌다.
- Mobile은 같은 정보를 세로 타임라인으로 배치하고 상태 전이 행동을 단계 본문 아래에 노출한다.
- 현재 `in_progress` 단계만 브랜드 테두리와 옅은 민트 표면으로 강조한다. `blocked`는 경고 아이콘과 상태 문구를 함께 표시한다.
- 체크리스트와 링크는 기존 Phase 5 조밀도·입력 컴포넌트를 재사용해 Task 화면과 조작 방식이 달라지지 않게 한다.

### 38.4 템플릿과 인스턴스 구분

- Workflow 템플릿 카드는 단계 수, 권장 시기와 복제·적용 행동을 표시하고 실행 상태 배지를 사용하지 않는다.
- 인스턴스 카드는 상태, 연결 Task, 진행률과 다음 행동을 표시한다. 템플릿과 동일한 표면을 사용해도 `템플릿` 또는 `실행` 맥락을 제목으로 명확히 구분한다.
- Phase 5 단일 업무 템플릿과 Workflow 템플릿을 같은 화면이나 같은 명칭으로 합치지 않는다.

### 38.5 반응형·접근성·개인정보

- Desktop 본문+오른쪽 패널은 1280px 미만에서 비율을 줄이고, 768px 미만에서는 단일 열과 접이식 템플릿 패널로 전환한다.
- 전이 버튼은 Desktop 최소 36px, Mobile 최소 44px 높이를 사용한다. 위험 행동인 취소는 기존 `danger` 규칙과 확인 과정을 따른다.
- 포커스 순서는 검색·필터 → 인스턴스 목록 → 단계 → 템플릿·시작 패널이며 확장 영역을 연 뒤 첫 입력으로 이동한다.
- 상태 아이콘에는 대체 레이블을 제공하고 진행률·타임라인 연결선이 없어도 순서와 상태를 이해할 수 있어야 한다.
- 학생 이름·학번·질병명·검진 결과처럼 민감정보 입력을 연상시키는 플레이스홀더나 예시는 사용하지 않는다. 대상은 범주와 비식별 수량으로 표현한다.

### 38.6 디자인 QA

- 모든 상태 조합과 허용 전이 버튼을 Light 테마, 키보드 포커스, 200% 확대에서 확인한다.
- 긴 한국어 단계명, 체크리스트 0개·다수, 보류 단계와 종료 인스턴스에서 레이아웃을 확인한다.
- 375px에서 가로 오버플로가 없고 다음 행동과 핵심 전이 버튼이 첫 화면 흐름에서 발견되는지 확인한다.
- Phase 5 템플릿·연간 업무·Task 카드의 토큰과 조밀도가 변경되지 않았는지 비교한다.

## 39. Phase 7 AI 업무 도우미 컴포넌트 규칙

### 39.1 역할과 상태

- AI 표면은 기존 `surface`, `line`, `radius-md`, `shadow-sm`을 사용하고 별도 제품처럼 보이는 그라데이션·과도한 광택·마법 효과를 사용하지 않는다.
- AI 결과에는 항상 `AI 제안` 레이블을 표시한다. mock 응답은 `예시 응답` 배지를 추가하며 실제 provider 결과와 같은 성공색을 사용하지 않는다.
- 상태는 `입력 전`, `요청 중`, `제안`, `확인 필요`, `반영 중`, `반영 완료`, `제한`, `오류` 텍스트를 아이콘과 함께 표시한다.

### 39.2 입력과 문맥

- 입력부 위 또는 바로 아래에 민감정보 금지 안내를 상시 표시한다. 안내를 placeholder에만 넣지 않는다.
- 선택 문맥 chip은 종류와 비식별 제목만 표시하며 닫기 버튼으로 요청 전 제거할 수 있다. 전체 목록 자동 첨부를 암시하는 표현을 사용하지 않는다.
- 입력 예시는 `다음 주 보건교육 결과 제출 업무와 체크리스트를 정리해 줘`처럼 개인을 식별하지 않는 문장만 사용한다.

### 39.3 제안과 diff

- 자유 형식 답변과 저장 가능한 구조화 제안을 시각적으로 분리한다. 저장 가능한 제안에는 대상 종류, 필드 diff와 `확인 전에는 저장되지 않음` 문구를 표시한다.
- 변경 전 값은 중성 표면, 제안 값은 옅은 브랜드 보조 표면을 사용하되 텍스트 레이블과 기호로도 구분한다.
- 위험색은 provider·검증 오류에만 사용한다. AI 제안 자체를 성공으로 표시하지 않으며 `확인하고 반영`이 유일한 강조 저장 행동이다.
- 삭제 행동과 휴지통 아이콘은 AI 제안 카드에 두지 않는다.

### 39.4 반응형·접근성

- 1280px 이상은 입력·응답·문맥의 3영역을 허용하고, 1280px 미만은 보조 문맥을 접으며, 768px 미만은 단일 열로 전환한다.
- Mobile 주요 행동은 최소 44px, Desktop은 최소 36px 높이를 사용한다. 긴 diff는 줄바꿈하고 표가 필요하면 필드별 세로 목록으로 변환한다.
- 요청 중 spinner만 표시하지 않고 상태 문구를 함께 제공한다. 결과 영역은 포커스를 강제로 빼앗지 않으며 완료 알림 후 사용자가 결과로 이동할 수 있게 한다.
- 제안 출처, mock 여부, 저장 여부와 오류는 색상만으로 구분하지 않는다. 200% 확대와 키보드·스크린 리더 흐름을 확인한다.

### 39.5 회귀 기준

- 기존 빠른 입력 미리보기, Task·Event 폼, Workflow 상태 전이와 전체 캘린더의 시각 계층을 변경하지 않는다.
- AI 진입점은 브리핑의 핵심 업무와 오늘 꼭 끝낼 일을 밀어내지 않으며 Mobile 하단 내비게이션의 기존 주요 항목을 숨기지 않는다.
