# Phase 7 AI Architecture

## 1. 목표와 경계

Phase 7 AI는 보건교사의 업무 문장을 정리하고 기존 Task·Event·Workflow 문맥에서 다음 행동이나 변경안을 제안하는 보조 계층이다. AI 응답은 데이터베이스 명령이 아니며, 사용자가 미리보기를 검토하고 명시적으로 확인하기 전에는 어떤 사용자 데이터도 변경하지 않는다.

Phase 1~6의 인증, Repository, 원자적 RPC, RLS와 업무 규칙을 그대로 사용한다. AI가 별도 권한으로 Supabase에 접근하거나 기존 검증을 우회하지 않는다. Phase 8 기능은 이 문서의 범위가 아니다.

## 2. 서버 전용 구성

다음 값은 모두 서버 런타임에서만 읽는다. `NEXT_PUBLIC_` 접두사를 사용하거나 브라우저 번들, HTML, API 응답, 로그에 값 자체를 노출하지 않는다.

```text
AI_PROVIDER=mock
OPENAI_API_KEY=
AI_MODEL=
```

- `AI_PROVIDER`: AI 구현 선택자다. 지원하지 않는 값은 안전하게 거부하고 임의 제공자로 전환하지 않는다.
- `OPENAI_API_KEY`: OpenAI provider에서만 필요하다. 저장소, 클라이언트 코드와 로그에 포함하지 않는다.
- `AI_MODEL`: 서버가 허용한 모델 식별자다. 클라이언트가 모델명을 지정하거나 덮어쓸 수 없다.
- 로컬 개발과 provider 장애 시 `mock` 구현을 사용할 수 있다. 운영에서 자격 증명이 없거나 provider 호출이 실패하면 mock 결과로 조용히 가장하지 않고, 사용자에게 제한된 대체 결과임을 표시한다.

## 3. Provider 추상화

UI와 route handler는 특정 SDK 응답 형식에 의존하지 않는다. 서버의 provider interface가 최소 문맥과 요청 목적을 받아 공통 구조를 반환한다.

```text
UI → 인증된 서버 endpoint → 입력·권한·속도 검증
   → AI provider interface → OpenAI provider 또는 mock provider
   → 구조 검증·허용 행동 필터 → 미리보기 응답
   → 사용자 확인 → 기존 Repository/RPC → RLS가 적용된 저장
```

공통 결과는 설명용 문장과 구조화된 제안 목록을 분리한다. 구조화된 제안은 런타임 스키마로 검증하며 알 수 없는 필드, 허용되지 않은 enum, 잘못된 날짜·시간과 과도한 문자열 길이를 거부한다. provider 원문 응답이나 자유 형식 JSON을 저장 함수에 직접 전달하지 않는다.

mock provider는 고정되고 재현 가능한 fixture를 반환하며 네트워크를 사용하지 않는다. 테스트는 mock을 기본으로 하고 실제 API 키나 외부 호출에 의존하지 않는다.

## 4. 허용 결과와 no-delete 계약

Phase 7에서 허용하는 결과는 다음 세 종류다.

- `answer`: 현재 화면의 최소 문맥을 바탕으로 한 읽기 전용 안내·요약
- `suggestion`: 우선순위, 일정, 체크리스트 또는 다음 행동에 대한 읽기 전용 제안
- `mutation_preview`: 기존 제품 폼으로 옮길 수 있는 Task·Event 생성 또는 허용 필드 수정안

AI 계약에는 delete operation이 존재하지 않는다. Task, Event, Workflow, 단계, 체크리스트, 템플릿과 기록의 삭제·취소·일괄 제거는 제안하거나 실행하지 않는다. provider가 삭제를 암시하거나 알 수 없는 operation을 반환하면 서버 검증에서 폐기한다. Workflow 상태 전이와 완료도 기존 화면의 명시적 행동과 RPC 규칙을 사용하며 AI가 직접 실행하지 않는다.

모든 변경안은 다음 순서를 지킨다.

1. AI가 현재 값과 제안 값을 구분한 구조화된 미리보기를 반환한다.
2. 사용자가 항목별로 포함 여부를 선택하고 기존 폼에서 수정할 수 있다.
3. 확인 화면은 생성·수정 대상과 실제 변경 필드를 다시 보여준다.
4. 사용자가 `확인하고 반영`을 누른 뒤에만 기존 server action과 Repository/RPC를 호출한다.
5. 저장 시 인증, 입력 검증, 소유권, 상태 전이와 RLS를 다시 확인한다. 실패하면 일부 성공처럼 표시하지 않는다.

## 5. 최소 문맥

서버는 요청 목적을 수행하는 데 필요한 필드만 provider에 보낸다.

- 기본: 사용자가 현재 입력한 문장, 요청 종류, 한국어 locale, `Asia/Seoul` 시간대와 오늘 날짜
- Task 문맥: 제목, 허용된 상태·카테고리·우선순위·날짜, 비식별 체크리스트와 현재 Workflow의 다음 행동
- Event 문맥: 제목, 시작·종료 일시, 종일 여부와 카테고리
- 목록 문맥: 사용자가 명시적으로 선택한 항목의 제한된 요약만 포함하며 전체 데이터베이스를 전달하지 않는다.

Supabase access token, Google token, 이메일, `user_id`, 내부 행 식별자, 링크 본문, 로그와 다른 사용자의 데이터는 provider에 보내지 않는다. 서버가 소유권 확인에 필요한 ID를 사용하더라도 provider payload에서는 제거한다. 메모·설명은 기본적으로 제외하고 사용자가 해당 텍스트를 명시적으로 AI 입력에 포함한 경우에만 전송한다.

## 6. 오류, 제한과 관측

- 인증 사용자 단위로 짧은 구간 요청 제한을 적용한다. 제품 기본값은 분당 10회이며 초과 시 재시도 가능 시점을 안내한다.
- 전체 요청은 12초에서 중단한다. timeout, 429, provider 장애와 잘못된 구조 응답은 서로 구분 가능한 내부 코드로 처리하되 사용자에게 키, 모델 원문 응답이나 stack trace를 노출하지 않는다.
- 자동 재시도는 timeout·429·5xx에 최대 1회만 적용하고, 변경 반영 요청은 자동 재시도하지 않는다.
- 로그에는 request ID, provider 종류, 성공 여부, 지연 시간, 오류 분류와 token 수처럼 비식별 운영 정보만 남긴다. 사용자 입력, provider 원문, 제안 본문과 API 키는 기록하지 않는다.
- mock fallback이 사용되면 응답에 `source: mock`과 제한 안내를 포함한다. mock 제안도 동일한 미리보기·확인 절차를 거친다.

## 7. 선택적 기록

대화·제안 기록 저장은 기본값이 꺼져 있다. 기본 모드에서는 요청 처리 후 사용자 입력과 AI 응답을 데이터베이스, localStorage 또는 분석 도구에 보관하지 않는다. 현재 패널이 열려 있는 동안의 화면 상태만 허용한다.

`20260718130000_phase_7_ai_data.sql`은 사용자별 `ai_preferences.history_enabled` 기본값을 `false`로 두고, opt-in 기록을 위한 `ai_requests`와 `ai_action_drafts`를 준비한다. 기록은 사용자가 설정과 해당 요청에서 명시적으로 선택한 경우에만 생성하며 `save_ai_history_bundle` RPC가 설정·요청·초안을 한 트랜잭션으로 저장한다. 문맥 스냅샷, 인증정보와 전체 화면 데이터는 저장하지 않는다.

세 테이블은 모두 `user_id`, 소유권 인덱스와 사용자별 CRUD RLS를 가지며 `anon` 접근을 허용하지 않는다. action draft는 같은 사용자의 request에만 연결한다. opt-out 이후에는 신규 기록 생성을 즉시 중단하고, 보존·조회·삭제 UX와 provider 측 보존 정책을 배포 전에 검증한다.

## 8. 검증 기준

- provider 교체가 UI와 저장 계층 변경 없이 가능하다.
- 세 서버 전용 환경 변수가 클라이언트 번들·응답·로그에 노출되지 않는다.
- mock과 OpenAI 응답이 같은 런타임 스키마와 no-delete 필터를 통과한다.
- 미리보기 취소, schema 오류, timeout과 rate limit에서 DB 쓰기가 0건이다.
- 확인 후 저장도 기존 사용자 A/B·비로그인 RLS와 상태 전이 검증을 통과한다.
- 기록 기본값이 꺼진 상태에서 새 대화·메시지 행과 localStorage 원본이 생성되지 않는다.
