# BOGUNON 화면 명세서

## 1. 문서 목적

이 문서는 BOGUNON의 PC·모바일 화면 구조, 화면별 목적, 구성 요소, 사용자 행동, 반응형 전환과 상태 표현을 정의한다.

화면 설계는 다음 원칙을 따른다.

- 첫 화면은 캘린더가 아니라 `오늘의 브리핑`이다.
- 일정관리 앱으로서 축소 월간 달력, 전체 월간 캘린더, 주간 캘린더와 일정 목록 보기를 제공한다.
- PC는 왼쪽 고정 사이드바와 중앙 작업 캔버스, 오른쪽 운영 열을 사용한다.
- 편집용 우측 상세 패널은 오른쪽 운영 열과 별개이며 필요한 순간에만 연다.
- 모바일은 주간 스트립과 오늘 행동을 우선한다.
- 모든 정보를 큰 흰색 카드로 감싸지 않는다.
- 학생 개인정보를 입력하거나 표시하는 화면을 만들지 않는다.

## 2. 화면 설계 원칙

### 2.1 브리핑 중심

오늘 꼭 끝낼 일, 회신 대기, 확인 필요, 오늘 일정과 운동을 캘린더보다 먼저 보여준다.

### 2.2 달력의 두 역할

- 축소 달력: 빠른 날짜 탐색과 날짜별 업무량 파악
- 전체 캘린더: 월·주 단위 조망, 필터와 상세 진입

두 달력은 같은 컴포넌트 크기만 바꿔 재사용하지 않는다. 날짜 계산과 항목 집계 로직은 공유할 수 있지만 정보 밀도와 상호작용은 각 목적에 맞게 설계한다.

### 2.3 선택 후 상세

긴 편집 폼을 브리핑이나 캘린더 본문에 직접 펼치지 않는다. 항목 선택 시 우측 상세 패널 또는 모바일 상세 화면을 사용한다.

### 2.4 정보 밀도

다음 표현을 혼합한다.

- 구분선 기반 목록
- 시간순 타임라인
- 밀도 있는 업무 행
- 상태 배지
- 숫자 요약
- 제한된 핵심 카드
- 상세 패널

## 3. 전체 내비게이션

### 3.1 PC 고정 사이드바

위쪽:

- BOGUNON 워드마크
- 브리핑
- 업무
- 업무 절차
- 캘린더
- 연간 계획
- 운동

아래쪽:

- 동기화 상태
- 설정
- 프로필과 로그아웃 메뉴

현재 메뉴는 왼쪽 인디케이터, 글자 굵기와 옅은 배경 차이로 표시한다. 빠른 추가는 사이드바 상단 버튼과 브리핑 오른쪽 운영 열에서 제공한다.

### 3.2 모바일 하단 메뉴

- 브리핑
- 업무함
- 일정
- 운동
- 설정

새 항목 생성은 화면 상단 우측의 `+ 새로 만들기`로 제공한다. 하단 메뉴 위에 떠 있는 대형 원형 버튼을 사용하지 않는다.

### 3.3 프로필 메뉴

프로필 아바타 선택 시 작은 메뉴를 연다.

- 계정 식별 정보
- 설정
- 로그아웃

로그인 후 큰 계정 연동 버튼을 상시 표시하지 않는다.

## 4. PC 공통 레이아웃

### 4.1 기본 프레임

- 왼쪽 사이드바: 데스크톱에서 화면 높이에 고정
- 중앙 작업 캔버스: 월간 통합 캘린더와 주간 요약
- 오른쪽 운영 열: 오늘 행동을 밀도 있게 상시 요약
- 우측 상세 패널: 선택 시에만 운영 열 위 오버레이로 표시

### 4.2 우측 상세 패널 동작

열리는 상황:

- 일정 상세
- 할 일 상세
- 항목 수정
- 빠른 생성
- 선택 날짜 상세

규칙:

- 기본 닫힘 상태다.
- 패널이 열려도 현재 목록 또는 달력의 선택 맥락을 유지한다.
- 닫기 버튼과 `Esc` 키를 지원한다.
- 열릴 때 첫 제목 또는 첫 입력 필드로 포커스를 이동한다.
- 닫을 때 이전 선택 요소로 포커스를 돌린다.
- 저장되지 않은 변경이 있으면 닫기 전에 확인한다.
- 좁은 화면에서는 전체 높이 오버레이로 전환한다.

## 5. PC 로그인 화면

### 목적

제품 성격과 개인정보 원칙을 설명하고 로그인을 시작한다.

### 구성

- BOGUNON 워드마크
- 설명: `오늘의 업무와 일정을 한곳에서 정리하세요.`
- `Google로 로그인` 버튼
- 개인정보 안내
- 로그인 오류 메시지

### 시각 구조

- 화면 중앙의 단일 로그인 영역
- 대시보드 미리보기처럼 보이는 장식용 캘린더를 배경에 사용하지 않음
- 테두리만 있는 계정 연동 배너 형태를 사용하지 않음

### 상태

- 기본
- 로그인 진행 중
- 로그인 취소
- 로그인 실패

## 6. PC 브리핑 화면

### 목적

오늘 행동, 기다리는 업무, 일정과 운동을 우선 확인하면서 다른 날짜로 빠르게 이동한다.

### 레이아웃

- 왼쪽 고정 사이드바: 224~248px
- 중앙 운영 캔버스: 남은 너비의 중심, 최소 600px
- 오른쪽 운영 열: 300~340px
- 운영 모듈은 큰 장식 카드보다 작은 제목, 구분선과 밀도 있는 행을 사용

### 중앙 상단

- 페이지 제목 `오늘의 브리핑`
- 날짜와 요일
- 현재 근무 상태
- 퇴근까지 남은 시간
- 규칙 기반 오늘 요약 문장

### 중앙 본문 순서

1. 이번 주 요약
2. 월간 통합 캘린더

### 오른쪽 운영 열

- 오늘의 업무
- 회신 대기
- 회의·일정
- 운동
- 빠른 추가

### 사용자 행동

- 할 일 완료 또는 완료 취소
- 상태 배지 선택 후 상태 변경
- 항목 선택 후 상세 패널 열기
- 월간 통합 캘린더의 날짜 흐름 확인
- 오늘로 이동
- 전체 캘린더 이동
- 새로 만들기 패널 열기

### 제한

- 긴 편집 폼을 직접 표시하지 않는다.
- 월간 달력은 일정·업무와 날짜 스티커를 함께 조망하되 오른쪽 행동 모듈보다 과도하게 장식하지 않는다.
- 날짜·시간·근무 상태는 중앙 헤더에 두고 사이드바에 중복하지 않는다.

## 7. 오늘 요약 문장

### 목적

저장된 데이터를 규칙적으로 조합해 오늘 상황을 한 문장 또는 두 문장으로 요약한다.

### 생성 우선순위

1. 오늘 꼭 끝낼 일 수
2. 회신 대기·확인 필요 수
3. 다음 일정
4. 오늘 운동

### 예시

- `오늘 꼭 끝낼 일이 3개 있습니다.`
- `회신 대기 2건, 확인 필요 1건이 있습니다.`
- `다음 일정은 오후 3시 교직원 회의입니다.`
- `오늘 운동은 오후 7시 배드민턴 레슨입니다.`

### 규칙

- AI를 사용하지 않는다.
- 해당 데이터가 없는 문장 조각은 생략한다.
- 개인을 식별하는 내용을 추가 생성하지 않는다.
- 최대 두 문장으로 제한한다.
- 요약할 데이터가 없으면 `오늘 등록된 일정과 핵심 업무가 없습니다.`를 표시한다.

## 8. 오늘 꼭 끝낼 일 섹션

### 구성

- 섹션 제목
- 지정 수 `2/3`
- 밀도 있는 할 일 행 최대 3개
- 전체 업무에서 선택하는 보조 행동

### 행 정보

- 완료 체크
- 제목
- 영역 라벨
- 우선순위
- 마감 표시
- 상태

완료 항목은 당일 취소선과 완료 표시를 유지한다.

## 9. 회신 대기·확인 필요 섹션

### 구성

- `회신 대기`와 `확인 필요` 분할 필터
- 각 수량
- 최대 5개 업무 행
- 전체 보기

### 행 정보

- 제목
- 상태 배지
- 후속 확인일
- 마감 상태
- 영역

동일 업무를 같은 목록에서 중복 표시하지 않는다.

## 10. 오늘 시간순 일정

### 표현

카드 그리드가 아니라 세로 타임라인을 사용한다.

### 항목

- 시간 또는 `종일`
- 제목
- 영역 라벨
- 일정 유형 아이콘
- 연결된 미완료 할 일 수

### 정렬

1. 종일 일정
2. 시작 시간이 빠른 일정
3. 시간이 없는 일정

현재 시각은 얇은 기준선과 `현재` 라벨로 표시할 수 있다. 민트 원형 날짜 표현과 결합하지 않는다.

## 11. 축소 월간 달력

### 목적

브리핑을 벗어나지 않고 날짜를 탐색하고 업무 밀도를 확인한다.

### 구성

- 월·연도 텍스트
- 이전 달·다음 달 버튼
- 월요일 시작 요일 행
- 6주를 수용하는 조밀한 날짜 그리드
- 오늘로 이동 버튼

### 날짜 셀

- 날짜 숫자
- 일정 존재 표시
- 마감 존재 표시
- 운동 존재 표시
- 최대 2개의 짧은 표시 또는 점
- 초과 수 `+3`

### 오늘 표시

- 날짜 숫자 옆의 작은 네이비 인디케이터 또는 얇은 밑줄
- 셀 전체를 민트 원형으로 채우지 않음

### 선택 날짜 표시

- 얇은 청록 테두리와 옅은 청록 배경
- 오늘 표시와 선택 표시를 동시에 구분할 수 있어야 함

### 사용자 행동

- 날짜 선택
- 이전·다음 달 이동
- 오늘로 이동
- 전체 캘린더 열기

### 제한

- 긴 제목을 여러 줄로 표시하지 않는다.
- 참고 화면과 유사한 넓고 높은 셀 비율을 사용하지 않는다.
- 달력 전체를 큰 둥근 흰색 카드 안에 과도한 여백으로 배치하지 않는다.

## 12. 선택 날짜 간단 목록

### 위치

PC 브리핑 월간 통합 캘린더의 선택 날짜 맥락에 표시한다.

### 구성

- 선택 날짜
- 일정·할 일·마감·운동 수량
- 최대 5개 항목

### 행 정보

- 마감 또는 상태 표시

항목 선택 시 우측 상세 패널을 연다.

## 15. PC 전체 캘린더

### 목적

월·주 단위의 전체 일정과 날짜별 항목을 조망한다.

### 상단 도구막대

- 페이지 제목 `캘린더`
- 이전·다음 기간
- 오늘
- 월간·주간·일정 목록 보기 전환
- 일정·업무·장소·날짜 스티커 검색
- `일정 종류` 필터: 전체·업무·학교·개인
- `스티커 표시` 필터: 전체와 실제 구현된 날짜 스티커 팩
- 필터와 구분된 `+ 날짜 스티커` 실행 버튼
- `+ 일정 추가`

일정 종류 필터는 Task/Event 범위만, 스티커 표시 필터는 날짜 스티커 표시 범위만 제어한다. 두 상태는 서로 변경하지 않는다. 데스크톱에서는 일정 필터와 추가 버튼을 같은 행에 두고 스티커 표시 필터를 다음 행에 둔다. 768px 이하에서는 충돌 없이 줄바꿈하고, 375px에서는 검색 → 일정 필터 → 전체 폭 추가 버튼 → 내부 가로 스크롤 스티커 필터 순으로 배치한다.

### 표시 대상

- 일정
- 수행일이 있는 할 일
- 마감일
- 운동 일정

### 월간 보기

- 중앙 캔버스를 충분히 사용하되 빈 셀을 과도하게 높이지 않는다.
- 셀에는 최대 2개의 짧은 항목과 초과 수를 표시한다.
- 날짜 선택 시 우측 상세 패널 또는 캘린더 아래 날짜 목록을 연다.
- 날짜 숫자, 영역 마커와 유형 아이콘을 함께 사용한다.
- 일정은 최대 2개, 학교 날짜 스티커는 1개, 운동 스티커는 최대 2개를 표시하고 나머지는 `+N`으로 합친다.

### 주간 보기

- 월요일부터 일요일까지 표시
- 시간 축과 종일 영역 분리
- 시간이 있는 일정은 시간 위치에 표시
- 수행일·마감일은 상단 업무 영역에 표시
- 학교 날짜 스티커와 운동 스티커를 의미가 다른 자산으로 함께 표시한다.

## 날짜 스티커 선택 패널: 학사일정·보건업무·공휴일 (2026-07-19)

- 날짜 스티커 패널 상단은 학교·학사일정·보건업무·공휴일·개인 5개 탭이다. 767px 이하에서는 탭 자체만 가로 스크롤하며 페이지 전체 가로 스크롤은 만들지 않는다. 활성 탭이 화면 밖에 있으면 탭 목록 안에서 보이도록 스크롤한다.
- 학사일정 탭은 검색 입력과 전체·학기·시험·행사·운영 필터를 제공하고 결과를 sortOrder 순 3열 모바일, 5열 데스크톱 grid로 표시한다.
- 보건업무 탭은 검색 입력과 전체·건강검사·보건교육·운영·점검·행정·협업 필터를 제공한다. 결과는 sortOrder 순으로 정렬하고 375px 모바일에서도 3열 grid, 데스크톱에서는 5열 grid를 사용한다.
- 공휴일 탭은 검색 입력과 전체·국가 공휴일·명절·대체·특별 휴일·일반 휴일 필터를 제공한다. 카테고리별 개수는 20/11/4/4/1이며 결과는 sortOrder 순 3열 모바일, 5열 데스크톱 grid로 표시한다.
- 보건업무 검색 대상은 label, keywords, category, pack이다. `검사`, `교육`, `점검`, `CPR`, `담임` 검색 QA를 통과해야 하며 검색 input의 브라우저 기본 clear 버튼과 앱 clear 버튼을 중복 표시하지 않는다.
- 공휴일 검색 대상은 label, keywords, category, pack이다. `새해`, `명절`, `휴일`, `크리스마스`, `한글`, `투표`, `석가탄신일` 검색 QA를 통과해야 하며 `석가탄신일`은 `부처님 오신 날`을 찾는다.
- 팩 탭 변경, 카테고리 변경 또는 검색 결과 변경으로 현재 선택한 스티커가 보이지 않으면 선택 상태를 초기화한다. 하단 저장 버튼의 label과 disabled 상태는 실제 화면에 보이는 선택 상태와 항상 일치해야 한다.
- 스티커 버튼은 최소 44px 터치 영역, 날짜와 label이 포함된 aria-label, aria-pressed와 체크 아이콘으로 선택 상태를 전달한다.
- 스티커 선택은 저장과 분리한다. 시작일·종료일과 스티커를 고른 뒤 하단 저장 버튼을 눌러야 기존 server action이 실행된다.
- 각 SVG는 작은 셀에서도 식별되는 민트·네이비 외곽선과 파스텔 보조색을 사용한다. 캘린더에서는 심볼만, 선택 패널에서는 심볼과 label을 함께 표시한다. 보건업무 SVG는 의료기관용처럼 무겁거나 공포감을 주지 않고 개인정보, 질병명, 검사 결과를 표현하지 않는다. 공휴일 SVG는 상세 국기, 정치 캠페인, 지배적인 종교 상징, 작은 한글 문구 없이 추상 모티프로 구분한다.
- 월간·주간 캘린더의 스티커 클릭은 날짜 선택 이벤트를 전파하지 않고 기존 관리 UI를 연다. 같은 날짜의 일정·업무·다른 스티커는 삭제하지 않는다.
- 한 날짜에는 여러 팩의 날짜 스티커가 함께 존재할 수 있으며 기존 최대 표시와 `+N` overflow 규칙을 재사용한다. 학사일정·보건업무·공휴일 스티커는 같은 non-personal 날짜 스티커 lane에 표시하고 개인 스티커 lane과 운동 스티커 흐름을 변경하지 않는다.
- 패널은 안내 → 가로 스크롤 팩 탭 → 날짜 범위와 기간 안내 → 검색 → 카테고리 → 내부 스크롤 grid → sticky footer 순이다. grid는 데스크톱 5열, 모바일 3열이며 긴 label은 두 줄까지 표시한다.
- 결과가 없으면 `검색 결과가 없습니다.` 안내와 `검색 초기화` 버튼을 표시한다. footer CTA는 선택 전 `스티커를 선택하세요`, 하루 선택 후 `{label} 추가`, 기간 선택 후 `{N}일간 추가`, 저장 중 `추가 중…`으로 실제 상태를 반영한다.
- 캘린더의 날짜 스티커 표시 필터는 전체·학교·학사일정·보건업무·공휴일·개인 순서다. 이 필터는 일정/업무 필터와 독립적으로 동작하며 공휴일만 선택해도 Task와 Event 필터 상태를 바꾸지 않는다.
- 공휴일 팩은 자동으로 달력 날짜를 채우지 않는다. 사용자는 날짜를 직접 고르고 저장해야 하며, 일요일 표시, 음력 계산, 대체공휴일 자동 산정, 선거일 자동 발견, 공공 API 불러오기, 일괄 등록, 빨간 날짜 숫자 표시는 화면 범위가 아니다.
- PC 드래그 또는 모든 화면의 `날짜 변경` 메뉴는 동일한 확인 패널을 연다.

### 날짜 이동

- 일정과 날짜가 있는 업무만 이동할 수 있다.
- 반복 항목은 `이 일정만`, `이 일정과 이후 일정`, `전체 반복 일정` 중 하나를 선택한다.
- 저장 전 현재 날짜와 새 날짜를 함께 표시한다.
- 저장에 실패하면 화면의 원래 날짜를 유지하고 오류를 알린다.
- 운동 스티커, 학교 날짜 스티커와 반복 규칙 원본 자체는 직접 드래그하지 않는다.

### 검색

- 검색 대상은 일정·업무 제목, 설명, 메모, 장소와 학교 날짜 스티커 이름이다.
- 결과에는 날짜, 시간, 영역과 유형을 표시한다.
- 결과를 선택하면 해당 월 또는 주로 이동하고 날짜와 항목을 강조한다.
- 빈 검색어는 기록하지 않으며 결과가 없으면 명시적인 빈 상태를 표시한다.

## 29. 공통 상태

### 29.1 빈 상태

- 짧은 설명
- 하나의 주요 행동
- 수달 캐릭터를 작게 사용할 수 있음
- 참고 화면과 유사한 문구를 사용하지 않음

예시:

- `오늘 지정한 핵심 업무가 없습니다.`
- `선택한 날짜에 등록된 항목이 없습니다.`
- `기다리는 회신이 없습니다.`
- `첫 운동 일정을 기록해 보세요.`

### 29.2 로딩

- 전체 화면 스피너 대신 실제 레이아웃과 비슷한 낮은 대비 스켈레톤 사용
- 짧은 상태 변경은 버튼 내부 진행 표시 사용
- 기존 데이터가 있으면 유지하고 갱신 중임을 표시

### 29.3 저장 중

- 상단 동기화 상태: `저장 중`
- 저장 버튼 비활성화 및 진행 표시
- 중복 제출 방지

### 29.4 저장 성공

## 33. Phase 5 화면

### 업무 템플릿

### 업무 상세

- Task 카드에 체크리스트 완료 수와 비율, 관련 링크와 알림 수를 표시한다.
- 편집 패널에서 체크리스트 추가·수정·삭제·순서 이동, 링크와 알림 설정을 함께 관리한다.
- Task·Event의 복제 메뉴는 날짜와 포함할 설명·메모·체크리스트·반복 설정을 사용자가 선택한다.

- 이전 연도, 현재 연도와 다음 연도 이동을 제공한다.

### 빠른 입력

- 새로 만들기 패널 상단에서 한국어 문장을 해석한다.
- 결과 카드에 종류·제목·날짜·시간을 먼저 표시하고 `폼에 반영` 전에는 어떤 데이터도 저장하지 않는다.

## 34. Phase 6 Workflow OS 화면

### 34.1 내비게이션과 진입

- PC 고정 사이드바와 모바일 내비게이션에서 `Workflow`를 선택하면 `/workflows`로 이동한다.
- 브리핑의 Workflow 요약과 Task의 연결 표시는 동일한 인스턴스로 이동한다.
- Phase 5 `/templates`와 `/annual`은 유지한다. `/templates`는 단일 업무 재사용, `/workflows`는 Task에 연결된 다단계 절차 실행을 담당한다.

#### 연간 플래너 실사용 보완

- Desktop은 12개월 3열 그리드, Tablet은 2열, Mobile은 가로 스크롤 없는 1열 목록으로 표시한다. 현재 월은 연민트 배경과 `이번 달` 텍스트 배지로 함께 구분한다.
- 월 카드는 핵심 제안 세 개를 먼저 표시하고 `더 보기`로 나머지를 펼친다. 항목은 설명, 권장 시기, 예상 시간, 체크리스트 수와 추가 상태를 제공한다.
- `업무로 추가`와 `일정으로 추가`는 기존 생성 Sheet를 열며 저장하지 않는다. 정확한 날짜가 없으면 해당 월만 안내하고 날짜 입력을 필수로 한다.
- PC의 빠른 복사는 연도 안의 날짜를 먼저 선택한 뒤 항목을 드롭하면 같은 생성 Sheet를 연다. 드래그를 사용할 수 없는 사용자는 각 항목의 버튼으로 같은 흐름을 사용할 수 있다.
- `내 업무 추가`는 월, 제목, Task/Event 구분, 예상 시간과 체크리스트를 저장한다. 첫 구현은 사용자 항목 추가를 지원하며 기본 항목 숨김과 순서 변경은 후속 개선으로 둔다.

### 34.2 Desktop `/workflows`

- 1280px 이상에서는 가운데 검색·필터·인스턴스 목록과 오른쪽 템플릿·시작 패널을 나란히 배치한다.
- 상단에는 Workflow·연결 업무·단계·메모 검색, 상태·카테고리·진행률 필터와 `보류 단계만` 필터를 제공한다.
- 인스턴스 카드는 상태, 이름, 연결 Task, `완료 단계/전체 단계`, 진행률과 다음 행동을 먼저 표시한다.
- 인스턴스 안의 세로 단계 목록은 단계 상태, 설명, 예상 시간, 허용 전이 행동과 확장 가능한 메모·내부 참고사항·담당자 표시·완료 조건·체크리스트·링크 편집을 제공한다.
- 오른쪽 패널에서 미완료 Task와 기본 또는 사용자 템플릿을 선택해 Workflow를 시작한다. 확인 전에는 인스턴스를 저장하지 않는다.
- 기본 템플릿은 `내 템플릿으로 복제`할 수 있고, 사용자 템플릿은 단계와 후속 규칙을 포함해 생성·편집한다.
- 상태 전이 버튼은 현재 상태에서 허용된 행동만 제공한다. 완료·취소는 확인 과정을 사용하고 종료 상태에서는 편집 행동을 숨긴다.

### 34.3 Mobile `/workflows`

- 375px에서는 검색·핵심 필터, 인스턴스 카드, Workflow 시작, 템플릿 순으로 단일 열에 배치한다.
- 템플릿 관리와 완료·취소 이력은 접이식 섹션 또는 필터로 분리해 진행 중 인스턴스를 먼저 보이게 한다.
- 단계 목록은 세로 타임라인을 유지하고 `시작`, `완료`, `보류`, `건너뛰기`, `이 단계로 복귀`를 허용 전이에 따라 표시한다.
- 긴 단계명·메모·체크리스트는 줄바꿈하며 가로 스크롤을 만들지 않는다. 다음 행동과 핵심 전이는 첫 화면 흐름에서 접근 가능해야 한다.

### 34.4 대시보드와 Task 통합

- PC 브리핑 오른쪽 운영 열과 모바일 브리핑 본문에 진행 중 수, 보류 수와 가장 가까운 다음 행동을 표시한다.
- 다음 행동은 `in_progress`, `blocked`, `pending` 우선순위를 사용하고 미완료 체크리스트가 있으면 그 수를 먼저 안내한다.
- Workflow에서 생성된 후속 Task에는 `Workflow` 출처와 인스턴스 링크를 표시한다. 기존 Task 정렬·검색·필터·완료 UX는 변경하지 않는다.
- Workflow를 완료하면 원본 Task도 완료된다. 반대로 Task만 완료해도 Workflow 단계를 자동 완료하지 않으며 사용자가 Workflow에서 명시적으로 상태를 전이한다.

### 34.5 저장·오류·개인정보 상태

- 시작 또는 전이 중에는 해당 행동만 비활성화하고 중복 제출을 막는다. 성공 후 서버 결과로 인스턴스, 타임라인, Task와 대시보드를 함께 갱신한다.
- 원자적 RPC가 실패하면 이전 상태를 유지하고 일부 단계나 Task가 저장된 것처럼 표시하지 않는다. 재시도 행동과 오류 메시지를 제공한다.
- 다른 기기에서 상태가 먼저 바뀐 충돌은 최신 상태를 다시 불러온 뒤 가능한 행동을 다시 계산한다.
- 빈 상태와 입력 예시는 학생 개인정보를 유도하지 않는다. 메모·내부 참고사항 주변에는 개인 식별정보와 학생별 건강정보를 기록하지 말라는 안내를 제공한다.

### 34.6 화면 QA

- 1440×900, 1280×800, 768×1024, 375×812에서 템플릿 복제·생성, Task 적용, 단계 전이, 보류·재개, 완료·취소와 이력 필터를 확인한다.
- 키보드만으로 검색·필터·단계·확장 편집·확인 과정을 조작하고 저장 후 포커스가 논리적으로 유지되는지 확인한다.
- 상태와 진행률이 색상에만 의존하지 않고 텍스트·아이콘과 함께 표시되는지 확인한다.
- 새로고침과 네트워크 실패 뒤 부분 저장이나 중복 단계·후속 Task가 생기지 않는지 확인한다.
- Phase 5 `/templates`, `/annual`, Task 생성·복제와 체크리스트 화면이 그대로 동작하는지 회귀 확인한다.

## 35. Phase 7 AI 업무 도우미 화면

### 35.1 진입과 정보 구조

- PC 사이드바의 `AI 업무 도우미`와 브리핑·빠른 추가·업무·Workflow·연간 업무의 문맥 버튼은 현재 화면 위에 공통 반응형 AI 패널을 연다.
- 브리핑 카드는 짧은 입력과 최근 응답이 아닌 예시 행동을 제공한다. 빠른 추가, Task 상세와 Workflow 상세의 `AI로 정리`는 해당 화면의 선택 항목만 문맥으로 전달한다.
- AI 기록은 기본 off이므로 과거 대화 목록을 기본 화면에 표시하지 않는다. 새로고침 후 이전 대화를 복원한다고 약속하지 않는다.
- 모든 진입점에 `학생 개인정보나 건강정보를 입력하지 마세요` 안내를 입력부 가까이에 표시한다.

### 35.2 Desktop·Tablet

- 1280px 이상에서는 AI 패널을 오른쪽 detail panel로 열어 현재 업무 화면을 함께 확인할 수 있게 한다. 패널 안은 개인정보 안내 → 입력·예시 → 기록 선택 → 응답·제안 순서다.
- 768~1279px에서는 입력과 응답을 본문 단일 열로 두고 문맥·안내를 접이식 보조 패널로 전환한다.
- 목적은 `업무 정리`, `일정 초안`, `체크리스트 제안`, `다음 행동`, `선택 항목 요약`처럼 결과를 예측할 수 있는 명칭을 사용한다.
- loading 중 입력 전체를 지우지 않고 현재 요청의 제출만 막는다. 12초 전체 timeout, rate limit, provider 장애와 mock 응답을 구분해 안내한다.

### 35.3 Mobile

- 375px에서는 AI detail panel을 전체 화면 sheet로 전환하고 개인정보 안내 → 입력·예시 → 기록 선택 → 선택 문맥 → 응답 → 미리보기·확인의 단일 열 순서를 사용한다.
- 하단 고정 행동이 콘텐츠와 키보드를 가리지 않게 하고, 주요 버튼은 최소 44px 높이와 전체 너비를 사용한다.
- 긴 한국어 입력·응답·필드 diff는 줄바꿈하며 가로 스크롤을 만들지 않는다. 문맥 항목은 접어서 입력 흐름을 우선한다.
- 화면 이탈 전에 아직 반영하지 않은 미리보기가 있으면 폐기 여부를 확인하되, 대화 저장을 켜도록 유도하지 않는다.

### 35.4 미리보기·확인·결과

- 답변과 제안은 `AI 제안`으로 표시해 저장된 사실이나 확정 일정처럼 보이지 않게 한다.
- 변경 미리보기는 `새 업무`, `새 일정`, `업무 수정`, `일정 수정` 대상을 명시하고 변경 전·후 값을 필드별로 표시한다.
- 사용자는 항목별 포함 여부를 해제하고 값을 수정할 수 있다. `다시 제안`, `취소`, `폼에서 검토`, `확인하고 반영`을 구분한다.
- 삭제·취소·Workflow 전이 버튼은 AI 결과에 표시하지 않는다. provider가 해당 행동을 반환하면 오류 상태로 폐기한다.
- 성공 후 생성·수정된 항목 링크를 제공한다. 실패하면 미리보기를 유지하되 저장되지 않았음을 명시하고 중복 반영을 막는다.

### 35.5 빈 상태·오류·접근성 QA

- 빈 상태 예시는 비식별 업무 문장만 사용하고 실제 학생 사례를 연상시키지 않는다.
- 키보드 포커스는 목적 → 문맥 → 입력 → 제출 → 응답 → 미리보기 필드 → 확인 순서로 이동한다. 응답 갱신은 `aria-live`로 과도하지 않게 알린다.
- 색상 외에 `제안`, `mock`, `확인 필요`, `반영 완료`, `오류` 텍스트와 아이콘을 함께 사용한다.
- 1440×900, 1280×800, 768×1024, 375×812에서 모든 진입점과 loading·preview·confirm·timeout·rate limit·mock 상태를 확인한다.
- history off, 미리보기 취소와 provider 오류에서 새 DB 행·localStorage 대화 원본·부분 저장이 없는지 확인한다.
# 모바일 일정 중심 홈과 운동 스티커 화면 (2026-07-18)

- 모바일 홈 상단은 앱명, 22~24px 날짜·시간과 오늘 일정 건수만 간결하게 표시한다. 정상 동기화 상태는 작은 점과 `동기화됨`으로만 표시한다.
- 모바일 본문 순서는 `다음 일정 → 오늘 할 일 → 오늘 일정 → 주간 날짜 → 오늘 요약 → 오늘의 운동 → 월간 캘린더`다.
- 플로팅 추가 메뉴의 `운동 스티커 붙이기`는 오늘 날짜의 3열 스티커 sheet를 열고 스티커 탭 즉시 저장한다.
- 운동 화면은 월 이동·오늘 이동·이번 달 운동일·연속일을 제공하는 스티커 캘린더가 중심이다. 셀은 최대 두 스티커와 `+N`을 표시한다.
- 설정 화면은 loading skeleton, 실제 폼, 실제 오류와 재시도 상태를 구분하고 비로그인일 때만 계정 연결 안내를 표시한다.

## 빠른 보건업무 화면 보완 (Phase 8 아님)

### 업무 화면

- 업무 목록 위에 `자주 하는 보건업무`와 `반복 업무를 빠르게 추가하세요.` 안내를 compact 영역으로 표시한다.
- 기본 화면은 공통 registry의 우선 항목 6개만 보여 주며 `전체 보기`에서 12개를 일지·기록, 안내·소식, 검진, 교육, 시설·물품, 보고·제출로 묶는다.
- 최근 사용 데이터가 있을 때만 최대 4개를 별도 줄에 표시한다. 프리셋 선택과 전체 보기 행동은 Client Component 내부에서 처리한다.
- 각 항목의 별 버튼으로 즐겨찾기를 전환하며 즐겨찾기 항목은 사용자 순서 안에서 먼저 표시한다. `편집`에서는 드래그 또는 위·아래 버튼으로 순서를 바꾸고 항목을 숨길 수 있다.
- `숨긴 프리셋 관리`에서 항목을 복원하고 `기본 순서로 복원`에서 즐겨찾기·순서·숨김을 초기화한다. 최근 사용은 초기화 대상이 아니다.

### 모바일 새로 만들기

- 기존 업무·일정·개인 일정·운동 스티커·날짜 스티커·업무 절차·빠른 메모·작성 도움 메뉴를 유지한다.
- sheet 상단에 우선 프리셋 6개를 2열 compact grid로 표시하고 같은 sheet 안의 `전체 보기`로 12개를 확인한다.
- 모바일 편집 모드는 위·아래 이동 버튼, 숨기기, 숨긴 프리셋 복원과 기본 순서 복원을 제공한다. 같은 계정 설정은 업무 화면과 연간 플래너에도 즉시 반영한다.
- 프리셋을 누르면 기존 생성 sheet로 전환하며 저장 전까지 DB를 쓰지 않는다. 버튼은 최소 44px 터치 영역과 명시적 aria-label을 가진다.

### 생성 폼과 반응형

- Task 프리셋은 반복, 수행 시간, 체크리스트와 수행일 기준 알림을 채우고 Event 프리셋은 업무 영역, 기본 소요 시간과 알림을 채운다.
- Desktop·Tablet·Mobile 모두 같은 registry와 생성 폼을 사용하며 전체 페이지 가로 스크롤을 만들지 않는다.
- localStorage는 Client Component에서 hydration 이후에만 읽고 쓰며 서버에서 함수 props나 브라우저 API를 전달하지 않는다.

## PWA 설치와 브랜드 화면 (Phase 8 아님)

- 로그인, PC 사이드바, 모바일 오늘 헤더는 민트 B 심볼과 `BOGUNON` 워드마크를 사용한다. 한글 제품명, 설명 문구와 의료·학교 상징을 로고에 포함하지 않는다.
- 설정의 `앱으로 설치하기` 카드는 Android 설치 프롬프트 또는 iOS 홈 화면 추가 절차 중 현재 환경에 맞는 정보만 표시한다. standalone 모드에서는 카드 자체를 표시하지 않는다.
- 모바일 standalone 화면은 `viewport-fit=cover`, 상·하단 safe area와 `100dvh`를 사용한다. 하단 내비게이션과 플로팅 버튼은 홈 인디케이터와 겹치지 않는다.
- 오프라인 안내는 BOGUNON 심볼, 연결 확인 문구와 연민트 캔버스만 표시하며 사용자 데이터의 오래된 사본을 렌더링하지 않는다.
- standalone 첫 실행 환영 Dialog는 한 화면, 한 문장, `시작하기` 버튼으로 끝나며 두 번째 실행부터 표시하지 않는다.
- 설정의 앱 정보 카드는 버전과 설치 상태만 간결하게 보여주고, Android 설치 버튼·iOS 홈 화면 추가 안내를 환경에 맞게 구분한다.
