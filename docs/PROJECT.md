# 보건온 프로젝트 정의서

## 1. 문서 목적

이 문서는 보건온의 제품 목적, 사용자, 핵심 원칙, MVP 범위와 기술 경계를 정의한다. 세부 기능은 `PRODUCT_SPEC.md`, 데이터 구조는 `DATA_MODEL.md`, 개발 규칙은 `AI_RULES.md`를 따른다.

## 2. 프로젝트 개요

- 제품명: 보건온
- 제품 유형: 모바일·PC 반응형 개인 통합 일정관리 웹앱
- 초기 사용자: 현직 보건교사 1명
- 인증: Google 계정을 사용하는 Supabase Auth
- 주 저장소: Supabase Postgres Database
- 접근 제어: 모든 사용자 소유 테이블의 Row Level Security(RLS)
- 동기화: 같은 Google 계정으로 로그인한 PC와 모바일 간 클라우드 데이터 동기화
- 기본 언어: 한국어
- 기본 시간대: `Asia/Seoul`
- 주 시작일: 월요일
- 배포: Vercel
- 기술: Next.js App Router, TypeScript, Tailwind CSS, shadcn/ui, Lucide Icons, Supabase

## 3. 제품 콘셉트

보건온은 보건교사의 학교 업무와 운동, 개인 일정, 개인 프로젝트를 한곳에서 기록하고 같은 Google 계정으로 PC와 모바일에서 이어서 관리할 수 있게 하는 개인 통합 데일리 운영 시스템이다.

## 4. 프로젝트 배경

보건교사 업무는 일정과 단순 완료형 할 일만으로 관리하기 어렵다. 공문, 검진, 연수, 보고와 안내 업무는 여러 단계를 거치며 다른 교직원, 보호자 또는 외부 기관의 회신과 제출을 기다리는 경우가 많다. 업무 외에도 운동, 개인 일정과 출간·강의·앱 개발 등의 프로젝트를 함께 관리해야 한다.

학교에서는 PC로 상세 업무를 정리하고, 이동 중이나 퇴근 후에는 모바일로 확인·수정한다. 브라우저 한 곳에만 데이터가 저장되면 실제 사용 흐름이 끊기므로 동일한 Google 계정으로 로그인한 기기에서 같은 데이터를 사용할 수 있어야 한다.

보건온은 학생 건강정보 시스템이 아니다. 학생 관련 내용은 특정 개인을 식별할 수 없는 업무 상태와 수량으로만 기록한다.

## 5. 문제 정의

1. 학교 업무, 운동, 개인 일정과 프로젝트가 여러 도구에 분산된다.
2. 완료·미완료만으로 회신 대기, 확인 필요와 후속 확인을 표현하기 어렵다.
3. 반복되는 보건업무를 매년 다시 작성해야 한다.
4. 캘린더만으로 오늘 반드시 끝낼 행동을 파악하기 어렵다.
5. 학교 PC와 모바일의 데이터가 분리되면 연속적인 관리가 어렵다.
6. 일반 메모 도구에 학생 개인정보를 입력할 위험이 있다.

## 6. 제품 목표

1. 앱을 연 뒤 10초 안에 오늘의 핵심 업무와 일정을 파악한다.
2. 오늘 꼭 끝낼 일을 날짜별 최대 3개까지 직접 지정한다.
3. 회신 대기, 확인 필요와 후속 확인일을 독립적으로 관리한다.
4. 보건업무·학교일정·운동·개인·프로젝트를 함께 조회하고 명확히 구분한다.
5. 반복 업무 템플릿을 검토 후 실제 업무로 복사한다.
6. Google 로그인으로 사용자를 식별한다.
7. 같은 계정으로 로그인한 PC와 모바일에서 동일한 데이터를 조회·수정한다.
8. 모든 사용자 소유 데이터에 `userId`를 연결하고 RLS로 소유권을 강제한다.
9. JSON 내보내기와 복원으로 사용자가 별도 백업을 보유할 수 있게 한다.
10. 학생 개인정보를 저장하지 않는 UX와 데이터 구조를 유지한다.

## 7. 사용자

### 7.1 초기 사용자

초기 사용자는 현직 보건교사 1명이다. 다중 계정이 존재하더라도 협업 서비스가 아니라 각 사용자가 자신의 데이터를 독립적으로 관리하는 구조다.

### 7.2 인증 사용자

- Google OAuth로 로그인한다.
- Supabase Auth의 사용자 ID를 데이터 소유권 기준으로 사용한다.
- 같은 Google 계정으로 로그인하면 기기와 관계없이 같은 데이터를 조회한다.
- 다른 사용자의 데이터는 UI, API와 직접 요청 모두에서 접근할 수 없어야 한다.

### 7.3 향후 확장

다른 보건교사에게 제공할 수 있는 구조는 유지하되 MVP에서 학교 조직, 관리자, 공유, 협업과 역할 권한은 구현하지 않는다.

## 8. 핵심 영역

모든 일정과 할 일은 다음 중 하나에 속한다.

- 보건업무
- 학교일정
- 운동
- 개인
- 프로젝트

빠른 추가에서 데이터 영역은 `프로젝트`를 사용한다. 화면에서 프로젝트 자체 생성과 구분할 필요가 있을 때 `프로젝트 업무`로 표시할 수 있다.

## 9. 핵심 용어

- 일정: 특정 날짜와 시간 또는 기간에 발생하는 항목
- 할 일: 사용자가 수행하고 상태를 변경하거나 완료하는 행동
- 수행일: 할 일을 수행하기로 정한 선택 날짜
- 마감일: 할 일을 끝내야 하는 날짜
- 후속 확인일: 회신·제출·처리 상태를 다시 확인하는 날짜
- 오늘 꼭 끝낼 일: 사용자가 날짜별 최대 3개까지 직접 지정한 할 일
- 회신 대기: 다른 사람 또는 기관의 회신·제출·처리를 기다리는 상태
- 확인 필요: 결과나 처리 여부를 다시 확인해야 하는 상태
- 반복 업무 템플릿: 검토 후 실제 할 일로 복사하는 보건업무 원본
- 다음 행동: 프로젝트를 진행하기 위해 다음으로 수행할 구체적 행동
- 빠른 메모: 아직 일정이나 할 일로 구조화하지 않은 짧은 기록

## 10. 제품 원칙

### 10.1 오늘의 실행 우선

월간 캘린더보다 오늘 꼭 끝낼 일, 회신 대기, 확인 필요와 후속 확인을 우선한다.

### 10.2 사용자 결정 우선

앱은 핵심 업무, 상태, 진행률과 다음 행동을 임의로 선택하거나 변경하지 않는다.

### 10.3 PC·모바일 연속성

PC와 모바일은 동일한 데이터와 용어를 사용한다. 화면 크기에 따라 우선순위와 배치만 달라진다.

### 10.4 클라우드 원본

Supabase Database를 사용자 데이터의 유일한 원본으로 사용한다. localStorage를 원본 저장소나 동기화 대기열로 사용하지 않는다.

localStorage 허용 용도:

- 임시 작성 중인 폼 초안
- 마지막으로 열어 본 화면
- 필터와 UI 설정 캐시

### 10.5 서버 강제 소유권

클라이언트 필터만으로 데이터 접근을 제한하지 않는다. 모든 사용자 소유 테이블에 RLS를 활성화하고 인증 사용자 ID와 행의 `user_id`가 같은 경우에만 조회·생성·수정·삭제를 허용한다.

### 10.6 개인정보 최소화

로그인 계정 정보와 학생 개인 건강정보를 구분한다. 인증을 위해 Google 계정을 사용하더라도 학생 이름, 학번, 질병명, 상담 내용, 검진 결과, 보호자 연락처와 투약 내용을 저장하지 않는다.

### 10.7 자동화보다 예측 가능성

사용자가 직접 반복 주기를 지정한 업무만 예측 가능한 날짜 규칙으로 자동 생성한다. Phase 7 AI 추천은 사실 데이터와 분리된 선택적 초안으로만 제공하며 자동 우선순위 변경이나 자동 저장을 하지 않는다. 프로젝트 진행률 자동 계산은 구현하지 않는다.

## 12. 인증·보안 원칙

- Google OAuth는 Supabase Auth를 통해 처리한다.
- Next.js App Router의 브라우저·서버 클라이언트를 분리한다.
- 브라우저에는 Supabase publishable key만 노출할 수 있다.
- secret key 또는 service role key를 클라이언트 번들에 포함하지 않는다.
- 세션은 공식 SSR 방식의 쿠키 기반 흐름을 따른다.
- 모든 노출 테이블에 RLS를 활성화한다.
- SELECT·INSERT·UPDATE·DELETE 정책을 명시한다.
- UPDATE는 `USING`과 `WITH CHECK`에서 모두 소유권을 확인한다.
- RLS 정책은 `TO authenticated`와 `(select auth.uid()) = user_id`를 함께 사용한다.
- `user_metadata`를 권한 판단에 사용하지 않는다.
- 사용자 제공 `userId`를 신뢰하지 않고 인증 세션의 사용자 ID를 사용한다.

## 14. MVP 제외 범위

- 이메일·비밀번호 로그인
- 다른 소셜 로그인 제공자
- 학교 조직과 관리자
- 사용자 간 데이터 공유·협업
- 학생 명단과 학생별 기록
- 파일 업로드와 Supabase Storage
- Google Calendar 연동
- 푸시·문자·이메일 알림
- AI 추천·분류·요약·자동 작성
- 사용자 설정 없는 반복 업무 자동 생성
- 실시간 공동 편집
- 복잡한 오프라인 동기화
- 결제와 구독
- 네이티브 모바일 앱

Google 로그인, Supabase Auth, Supabase Database, RLS와 다중 기기 데이터 동기화는 제외 범위가 아니라 MVP 필수 범위다.

## 15. 성공 기준

- 같은 Google 계정으로 PC와 모바일에서 로그인할 수 있다.
- 한 기기에서 저장한 데이터가 다른 기기의 재조회에 나타난다.
- 다른 사용자 계정으로는 해당 데이터를 조회·수정·삭제할 수 없다.
- 비로그인 사용자는 보호된 사용자 데이터에 접근할 수 없다.
- 로그아웃해도 서버 데이터는 유지되고 다시 로그인하면 복원된다.
- localStorage를 삭제해도 서버 데이터는 사라지지 않는다.
- 학생 개인정보용 필드가 존재하지 않는다.
- 핵심 일정·업무·운동·프로젝트 기능과 JSON 백업이 동작한다.

## 16. 범위 변경 원칙

기능 추가 또는 제거는 사용자의 명시적 지시, 개인정보 검토, 문서 동기화와 데이터·보안 영향 검토 후 진행한다. Codex는 일반적인 서비스 관행이나 향후 가능성을 이유로 기능을 임의 추가하지 않는다.
