# 보건온 제품 기능 명세서

## 1. 문서 목적

이 문서는 보건온 MVP의 인증, 클라우드 저장, 동기화와 일정관리 기능의 입력값, 동작, 제한, 성공·오류 상태와 인수 조건을 정의한다. `PROJECT.md`와 `AI_RULES.md`를 상위 기준으로 사용한다.

## 2. 제품 범위

### 2.1 MVP 필수 기반

- Supabase Auth Google 로그인
- 쿠키 기반 세션 유지
- Supabase Database 주 저장소
- 모든 사용자 소유 데이터의 `userId`
- 모든 사용자 소유 테이블의 RLS
- 같은 계정으로 로그인한 PC·모바일의 동일 데이터
- 서버 저장 후 다른 기기 재조회 시 최신 데이터 반영

### 2.2 제품 기능

- 오늘 통합 대시보드
- 일정과 할 일
- 오늘 꼭 끝낼 일 최대 3개
- 회신 대기·확인 필요·후속 확인
- 월간·주간 캘린더
- 반복 업무 템플릿
- 운동 일정과 기록
- 프로젝트와 대표 다음 행동
- 빠른 메모
- 근무시간·교시 설정
- JSON 백업·복원

### 2.3 localStorage 범위

허용:

- 저장 전 폼 초안
- 마지막 화면
- 필터와 UI 설정 캐시

금지:

- 도메인 데이터 원본
- 서버 데이터 전체 복제
- 오프라인 쓰기 대기열
- 자체 세션 토큰 복제본

## 3. 인증 기능 명세

### 3.1 로그인 화면

표시 요소:

- 보건온 제품명과 간단한 설명
- `Google로 로그인` 버튼
- 학생 개인정보 저장 금지 안내
- 로그인 오류 영역

로그인 전에는 대시보드, 업무, 캘린더, 운동, 프로젝트와 설정 데이터에 접근할 수 없다.

### 3.2 Google 로그인 흐름

1. 사용자가 `Google로 로그인`을 선택한다.
2. 앱이 Supabase Auth Google OAuth를 시작한다.
3. 사용자가 Google에서 계정을 선택하고 인증한다.
4. Google이 설정된 OAuth 콜백으로 이동한다.
5. 앱이 인증 코드를 세션으로 교환한다.
6. 인증 사용자 ID를 확인한다.
7. 원래 요청한 보호 화면 또는 오늘 화면으로 이동한다.
8. 사용자 소유 데이터를 서버에서 조회한다.

### 3.3 로그인 성공

- 보호 화면을 표시한다.
- 현재 사용자의 데이터만 조회한다.
- 첫 로그인이고 설정 데이터가 없으면 현재 사용자 소유의 기본 설정 행을 생성한다.
- 다른 계정의 캐시나 폼 초안을 표시하지 않는다.

### 3.4 로그인 취소·실패

- 로그인 화면에 머문다.
- 사용자 데이터 요청을 실행하지 않는다.
- 메시지: `Google 로그인을 완료하지 못했습니다. 다시 시도해 주세요.`
- 재시도 버튼을 제공한다.

### 3.5 세션 만료

- 보호된 요청이 인증되지 않은 상태로 판정되면 저장 성공으로 표시하지 않는다.
- 현재 작성 중인 폼은 사용자별 임시 초안으로 보존할 수 있다.
- 로그인 화면으로 이동하고 `로그인이 만료되었습니다. 다시 로그인해 주세요.`를 표시한다.
- 다시 같은 계정으로 로그인하면 서버 데이터를 재조회한다.

### 3.6 로그아웃

1. 사용자가 로그아웃을 선택한다.
2. 확인 후 Supabase Auth 세션을 종료한다.
3. 메모리의 사용자 데이터와 사용자별 폼 초안을 제거한다.
4. 로그인 화면으로 이동한다.

로그아웃은 서버 데이터를 삭제하지 않는다.

## 4. 사용자 데이터 소유권

- 일정, 할 일, 오늘 꼭 끝낼 일 지정, 반복 업무 템플릿, 운동 기록, 프로젝트, 빠른 메모와 사용자 설정은 모두 인증 사용자 소유다.
- 각 행은 Supabase Auth 사용자 UUID와 연결된 `userId`를 가진다.
- 생성 시 `userId`는 현재 인증 사용자로 고정한다.
- 사용자는 `userId`를 화면에서 입력하거나 수정할 수 없다.
- 모든 CRUD는 RLS로 현재 사용자 소유 행에만 허용한다.
- 클라이언트에서 다른 사용자 ID를 전달해도 RLS가 거부해야 한다.

## 5. PC·모바일 동기화

### 5.1 기본 동작

- 서버 데이터가 원본이다.
- 앱 시작, 로그인 완료, 화면 재진입과 명시적 다시 불러오기에서 서버 데이터를 조회한다.
- 한 기기에서 저장 성공 후 다른 기기에서 재조회하면 최신 데이터가 나타나야 한다.
- 네트워크 연결이 있는 상태의 저장만 성공으로 처리한다.

### 5.2 저장 중 표시

- 서버 요청 중 저장 버튼을 중복 실행하지 못하게 한다.
- 저장 완료 전 화면을 닫으려 하면 입력 유지 여부를 확인한다.
- 성공: `[대상]을 저장했습니다.`
- 실패: `[대상]을 저장하지 못했습니다. 입력한 내용은 유지됩니다.`

### 5.3 수정 충돌

- 편집 시작 시 대상의 `updatedAt`을 보관한다.
- 저장 시 서버의 현재 `updatedAt`과 비교한다.
- 서버 값이 더 최근이면 자동 덮어쓰기하지 않는다.
- 사용자에게 `다른 기기에서 변경된 항목입니다.`를 표시한다.
- 선택:
  - `최신 내용 다시 불러오기`
  - `현재 입력으로 교체`
- 교체도 현재 사용자 소유권과 최신 버전을 다시 검증한다.

### 5.4 네트워크 오류

- 서버 저장에 실패하면 성공 상태를 표시하지 않는다.
- 원본 서버 데이터를 변경한 것으로 간주하지 않는다.
- 입력 폼을 유지하고 재시도를 제공한다.
- 오프라인 쓰기 큐는 만들지 않는다.

## 6. 공통 생성·수정·삭제 흐름

### 6.1 생성

1. 입력값 검증
2. 인증 사용자 확인
3. 새 ID 생성
4. `userId`를 인증 사용자 ID로 설정
5. 서버 저장
6. 서버 응답을 화면에 반영

### 6.2 수정

- `userId`, `id`, `createdAt`은 사용자가 변경할 수 없다.
- 저장 전 버전 충돌을 확인한다.
- 서버 성공 후 `updatedAt`을 갱신한다.

### 6.3 삭제

- 확인 대화상자 후 실행한다.
- RLS로 본인 행만 삭제할 수 있다.
- 관련 참조는 `DATA_MODEL.md`의 삭제 규칙에 따라 같은 사용자 범위에서 정리한다.
- 실패 시 기존 화면 데이터를 삭제된 것으로 표시하지 않는다.

## 7. 오늘 화면

모바일 순서:

1. 날짜·근무 상태
2. 오늘 꼭 끝낼 일
3. 회신 대기·확인 필요
4. 오늘 일정
5. 오늘 운동
6. 다음 개인 일정

PC 중앙 순서:

1. 오늘 꼭 끝낼 일
2. 회신 대기·확인 필요
3. 이번 주 요약
4. 월간 캘린더
5. 마감 예정

PC 오른쪽 패널:

- 현재 교시 또는 근무 상태
- 퇴근까지 남은 시간
- 다음 일정
- 오늘 운동
- 빠른 추가

표시 데이터는 현재 사용자 소유 데이터로 제한한다.

## 8. 일정

필수:

- 제목
- 영역
- 시작 날짜

선택:

- 종료 날짜
- 종일 여부
- 시작·종료 시간
- 메모
- 관련 링크
- 연결된 할 일
- 운동 일정 전용 정보

규칙:

- 종료 날짜는 시작 날짜보다 빠를 수 없다.
- 같은 날짜의 종료 시간은 시작 시간보다 늦어야 한다.
- 일정 하나에 현재 사용자의 여러 할 일을 연결할 수 있다.
- 할 일은 현재 사용자의 대표 일정 하나에만 연결할 수 있다.
- 일정 삭제 시 할 일은 유지하고 연결만 해제한다.
- 다른 사용자의 할 일 ID는 연결할 수 없다.

## 9. 할 일

필수:

- 제목
- 영역
- 상태
- 우선순위

선택:

- 수행일
- 마감일
- 후속 확인일
- 업무 단계
- 업무 대상
- 체크리스트
- 태그
- 색상
- 관련 링크
- 메모
- 대표 일정
- 프로젝트

상태:

- 예정
- 진행 중
- 회신 대기
- 확인 필요
- 완료
- 보류

우선순위:

- 낮음
- 보통
- 높음

기본 우선순위는 `보통`이다.

마감 표시:

- 마감일 이전 1~3일: 마감 임박
- 마감일이 오늘: 오늘 마감
- 마감일이 지남: 기한 경과
- 완료·보류: 마감 표시 제외

앱은 상태와 우선순위를 자동 변경하지 않는다.

## 10. 오늘 꼭 끝낼 일

- 날짜별 별도 컬렉션으로 관리한다.
- 현재 사용자의 할 일만 지정할 수 있다.
- 날짜별 최대 3개다.
- 완료된 항목은 당일 취소선과 함께 유지한다.
- 다음 날 자동 이월하지 않는다.
- 앱이 자동 선택·교체하지 않는다.

## 11. 회신 대기·확인 필요·후속 확인

- 회신 대기와 확인 필요는 저장 상태다.
- 후속 확인일은 마감일과 별도다.
- 오늘 또는 지난 후속 확인일의 미완료·비보류 할 일을 오늘 화면에 표시한다.
- 후속 확인일 도달만으로 상태를 변경하지 않는다.
- 같은 할 일이 여러 조건을 만족해도 같은 목록에서 한 번만 표시한다.

## 12. 캘린더

- 월간·주간 보기
- 월요일 시작
- 일정, 수행일이 있는 할 일과 마감일 표시
- 5개 영역 필터
- 현재 사용자 데이터만 표시
- 날짜 선택 후 빠른 추가
- 드래그 편집과 외부 캘린더는 제외

## 13. 반복 업무 템플릿

- 보건업무로 고정한다.
- 업무명, 체크리스트, 예상 시작 시점, 마감일 기준, 대상, 링크와 기본 메모를 저장한다.
- 현재 사용자 소유 템플릿만 조회·복사한다.
- 자동 생성하지 않는다.
- 적용 연도와 날짜를 검토한 뒤 새 할 일로 복사한다.
- 템플릿 삭제가 기존 복사 업무를 삭제하지 않는다.

## 14. 운동

- 운동 일정과 운동 기록을 관리한다.
- 운동 일정: 제목, 날짜, 시간, 종류, 장소, 목표와 메모
- 운동 기록: 완료 여부, 실제 운동일, 강도, 컨디션과 메모
- 운동 일정 하나에 기록 하나만 연결한다.
- 운동 일정과 기록의 `userId`가 같아야 한다.
- 운동 일정 삭제 시 연결 기록도 삭제한다.
- 운동 완료가 할 일이나 프로젝트를 변경하지 않는다.

## 15. 프로젝트

- 프로젝트명과 0~100 정수 진행률은 필수다.
- 진행률은 사용자가 직접 입력한다.
- 프로젝트당 대표 다음 행동 하나를 지정한다.
- 연결된 미완료 할 일 또는 직접 입력 텍스트 중 하나만 사용한다.
- 직접 입력은 할 일을 자동 생성하지 않는다.
- 프로젝트와 연결 할 일은 같은 사용자 소유여야 한다.
- 프로젝트 삭제 시 할 일은 유지하고 연결만 해제한다.

## 16. 빠른 메모

- 내용만 저장한다.
- 상태, 영역, 마감일과 완료 여부를 저장하지 않는다.
- 현재 사용자 소유 데이터다.
- 일정 또는 할 일로 전환할 수 있다.
- 전환 성공 후 원본 삭제 여부를 사용자가 선택한다.

## 17. 근무시간·교시 설정

- 사용자별 요일 근무 여부, 출퇴근 시간과 시간 구간을 저장한다.
- 같은 계정으로 로그인한 기기에서 같은 설정을 사용한다.
- 겹치는 시간 구간은 저장할 수 없다.
- 현재 교시와 퇴근까지 남은 시간은 계산하며 저장하지 않는다.

## 18. JSON 백업·복원

- 현재 로그인 사용자의 도메인 데이터만 내보낸다.
- 다른 사용자의 데이터와 인증 세션은 포함하지 않는다.
- 백업에 `ownerUserId`를 포함해 소유 사용자를 식별한다.
- 복원 시 로그인 사용자와 백업 소유자가 다르면 적용하지 않는다.
- 합치기는 같은 데이터 유형·동일 ID에서 최신 `updatedAt`을 유지한다.
- ID가 다르면 제목과 날짜가 같아도 별도 데이터다.
- 복원 행의 `userId`는 현재 인증 사용자로 검증하며 임의 변경을 허용하지 않는다.
- 전체 검증 성공 후 서버 트랜잭션 또는 실패 시 롤백 가능한 절차로 적용한다.

## 19. 개인정보 보호 UX

보건업무, 체크리스트, 빠른 메모, 템플릿과 백업 화면에 다음 안내를 표시한다.

> 학생 이름, 학번, 질병명, 상담 내용 등 개인 건강정보를 입력하지 마세요. 업무 상태와 비식별 수량만 기록해 주세요.

Google 로그인은 개인 데이터 동기화를 위한 것이며 학생정보 저장 허용을 의미하지 않는다.

## 20. 빈 상태

- 미로그인: `Google 계정으로 로그인해 보건온을 시작하세요.`
- 오늘 꼭 끝낼 일 없음: `오늘 꼭 끝낼 일로 지정한 항목이 없습니다.`
- 일정 없음: `오늘 예정된 일정이 없습니다.`
- 회신 대기 없음: `회신을 기다리는 업무가 없습니다.`
- 운동 없음: `오늘 예정된 운동이 없습니다.`
- 프로젝트 없음: `진행 중인 프로젝트가 없습니다.`
- 동기화 데이터 없음: 첫 데이터 추가 행동을 제공한다.

## 21. 오류 상태

- 인증 실패: `Google 로그인을 완료하지 못했습니다. 다시 시도해 주세요.`
- 세션 만료: `로그인이 만료되었습니다. 다시 로그인해 주세요.`
- 조회 실패: `데이터를 불러오지 못했습니다. 다시 시도해 주세요.`
- 저장 실패: `저장하지 못했습니다. 입력한 내용은 유지됩니다.`
- 권한 오류: `이 데이터에 접근할 수 없습니다.`
- 충돌: `다른 기기에서 변경된 항목입니다.`
- 네트워크 오류: `인터넷 연결을 확인한 뒤 다시 시도해 주세요.`

오류 시 다른 사용자의 데이터, 행 ID, SQL과 내부 정책명을 표시하지 않는다.

## 22. 확인 대화상자

- 일정·할 일·프로젝트·템플릿·운동·메모 삭제
- 미완료 체크리스트가 있는 할 일 완료
- 대표 연결 변경
- 백업 합치기·교체
- 수정 충돌 덮어쓰기
- 로그아웃

삭제와 교체는 결과를 구체적으로 설명하고 기본 포커스를 취소에 둔다.

## 23. 반응형 동작

모바일 하단 메뉴:

- 오늘
- 업무
- 캘린더
- 운동
- 나

플로팅 추가 버튼은 하단 메뉴와 콘텐츠를 가리지 않는다. PC는 고정 사이드바, 중앙 작업 영역과 오른쪽 오늘 패널을 사용한다. 인증 상태, 저장 중과 동기화 오류는 양쪽에서 같은 의미로 표시한다.

- 사용자 간 공유·협업
- 관리자·학교 조직
- 학생 명단과 학생별 건강기록
- Supabase Storage와 실제 첨부 업로드
- Google Calendar
- 푸시·문자·이메일 알림

- 자동 반복 업무
- 실시간 공동 편집
- 복잡한 오프라인 동기화
- 결제·구독

## 25. 인수 조건

### 인증·보안

- [ ] Google 로그인과 로그아웃이 동작한다.
- [ ] 비로그인 사용자는 보호 화면과 데이터에 접근하지 못한다.
- [ ] 사용자 A는 자기 데이터 CRUD가 가능하다.
- [ ] 사용자 A는 사용자 B의 행을 조회·생성·수정·삭제할 수 없다.
- [ ] UPDATE로 `userId`를 변경할 수 없다.
- [ ] 모든 사용자 소유 테이블에 RLS가 활성화된다.
- [ ] service role 또는 secret key가 브라우저에 노출되지 않는다.

### 동기화

- [ ] PC에서 저장한 데이터가 같은 계정의 모바일 재조회에 나타난다.
- [ ] 모바일 수정이 PC 재조회에 나타난다.
- [ ] localStorage를 삭제해도 서버 데이터가 유지된다.
- [ ] 저장 실패를 성공으로 표시하지 않는다.
- [ ] 더 최신 서버 데이터가 있으면 자동 덮어쓰기하지 않는다.

### 기존 제품 기능

- [ ] 일정·할 일·오늘 꼭 끝낼 일·후속 확인 규칙이 유지된다.
- [ ] 반복 템플릿은 검토 후 복사된다.
- [ ] 운동 기록과 프로젝트 진행률 규칙이 유지된다.
- [ ] JSON 백업은 현재 사용자 데이터만 포함한다.
- [ ] 학생 개인정보용 필드와 예시가 없다.

## 26. 예외 상황

- OAuth 팝업 또는 이동 취소: 로그인 화면 유지
- 다른 Google 계정 로그인: 해당 계정의 별도 데이터 조회
- 세션 만료 중 저장: 입력 유지 후 재로그인 요구
- 네트워크 단절 중 저장: 서버 미반영, 재시도 제공
- 두 기기 동시 수정: `updatedAt` 충돌 처리
- 로그아웃 후 뒤로 가기: 보호 화면 재검증
- RLS 거부: 데이터 미노출, 일반 권한 오류 표시
- 삭제된 연결 대상: 남은 데이터 유지, 연결만 해제
- 백업 소유자 불일치: 복원 차단

## 27. 미결정 사항

기능 범위는 확정됐다. 후속 문서에서 다음 구현 세부사항만 확정한다.

- Supabase 테이블·migration 파일 구조
- OAuth callback URL과 배포 환경 설정
- 동기화 재조회 시점의 정확한 UI 표시
- 충돌 감지 쿼리와 원자적 참조 처리 방식
- RLS·통합 테스트 실행 명령
