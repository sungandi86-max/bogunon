# BOGUNON 제품 기능 명세서

## 1. 문서 목적

이 문서는 BOGUNON MVP의 인증, 클라우드 저장, 동기화와 일정관리 기능의 입력값, 동작, 제한, 성공·오류 상태와 인수 조건을 정의한다. `PROJECT.md`와 `AI_RULES.md`를 상위 기준으로 사용한다.

## 2. 제품 범위

### 2.1 MVP 필수 기반

- Supabase Auth Google 로그인
- 쿠키 기반 세션 유지
- Supabase Database 주 저장소
- 모든 사용자 소유 데이터의 `userId`
- 모든 사용자 소유 테이블의 RLS
- 같은 계정으로 로그인한 PC·모바일의 동일 데이터
- 서버 저장 후 다른 기기 재조회 시 최신 데이터 반영

### 2.2 제품 기능

- 오늘 통합 대시보드
- 일정과 할 일
- 오늘 꼭 끝낼 일 최대 3개
- 회신 대기·확인 필요·후속 확인
- 월간·주간 캘린더
- 반복 업무 템플릿
- 운동 일정과 기록
- 빠른 메모
- 근무시간·교시 설정
- JSON 백업·복원

### 2.3 localStorage 범위

허용:

- 저장 전 폼 초안
- 마지막 화면
- 필터와 UI 설정 캐시

금지:

- 도메인 데이터 원본
- 서버 데이터 전체 복제
- 오프라인 쓰기 대기열
- 자체 세션 토큰 복제본

## 3. 인증 기능 명세

### 3.1 로그인 화면

표시 요소:

- BOGUNON 제품명과 간단한 설명
- `Google로 로그인` 버튼
- 학생 개인정보 저장 금지 안내
- 로그인 오류 영역

로그인 전에는 대시보드, 업무, 캘린더, 운동과 설정 데이터에 접근할 수 없다.

### 3.2 Google 로그인 흐름

1. 사용자가 `Google로 로그인`을 선택한다.
2. 앱이 Supabase Auth Google OAuth를 시작한다.
3. 사용자가 Google에서 계정을 선택하고 인증한다.
4. Google이 설정된 OAuth 콜백으로 이동한다.
5. 앱이 인증 코드를 세션으로 교환한다.
6. 인증 사용자 ID를 확인한다.
7. 원래 요청한 보호 화면 또는 오늘 화면으로 이동한다.
8. 사용자 소유 데이터를 서버에서 조회한다.

### 3.3 로그인 성공

- 보호 화면을 표시한다.
- 현재 사용자의 데이터만 조회한다.
- 첫 로그인이고 설정 데이터가 없으면 현재 사용자 소유의 기본 설정 행을 생성한다.
- 다른 계정의 캐시나 폼 초안을 표시하지 않는다.

### 3.4 로그인 취소·실패

- 로그인 화면에 머문다.
- 사용자 데이터 요청을 실행하지 않는다.
- 메시지: `Google 로그인을 완료하지 못했습니다. 다시 시도해 주세요.`
- 재시도 버튼을 제공한다.

### 3.5 세션 만료

- 보호된 요청이 인증되지 않은 상태로 판정되면 저장 성공으로 표시하지 않는다.
- 현재 작성 중인 폼은 사용자별 임시 초안으로 보존할 수 있다.
- 로그인 화면으로 이동하고 `로그인이 만료되었습니다. 다시 로그인해 주세요.`를 표시한다.
- 다시 같은 계정으로 로그인하면 서버 데이터를 재조회한다.

### 3.6 로그아웃

1. 사용자가 로그아웃을 선택한다.
2. 확인 후 Supabase Auth 세션을 종료한다.
3. 메모리의 사용자 데이터와 사용자별 폼 초안을 제거한다.
4. 로그인 화면으로 이동한다.

로그아웃은 서버 데이터를 삭제하지 않는다.

## 4. 사용자 데이터 소유권

- 일정, 할 일, 오늘 꼭 끝낼 일 지정, 반복 업무 템플릿, 운동 기록, 빠른 메모와 사용자 설정은 모두 인증 사용자 소유다.
- 각 행은 Supabase Auth 사용자 UUID와 연결된 `userId`를 가진다.
- 생성 시 `userId`는 현재 인증 사용자로 고정한다.
- 사용자는 `userId`를 화면에서 입력하거나 수정할 수 없다.
- 모든 CRUD는 RLS로 현재 사용자 소유 행에만 허용한다.
- 클라이언트에서 다른 사용자 ID를 전달해도 RLS가 거부해야 한다.

## 5. PC·모바일 동기화

### 5.1 기본 동작

- 서버 데이터가 원본이다.
- 앱 시작, 로그인 완료, 화면 재진입과 명시적 다시 불러오기에서 서버 데이터를 조회한다.
- 한 기기에서 저장 성공 후 다른 기기에서 재조회하면 최신 데이터가 나타나야 한다.
- 네트워크 연결이 있는 상태의 저장만 성공으로 처리한다.

### 5.2 저장 중 표시

- 서버 요청 중 저장 버튼을 중복 실행하지 못하게 한다.
- 저장 완료 전 화면을 닫으려 하면 입력 유지 여부를 확인한다.
- 성공: `[대상]을 저장했습니다.`
- 실패: `[대상]을 저장하지 못했습니다. 입력한 내용은 유지됩니다.`

### 5.3 수정 충돌

- 편집 시작 시 대상의 `updatedAt`을 보관한다.
- 저장 시 서버의 현재 `updatedAt`과 비교한다.
- 서버 값이 더 최근이면 자동 덮어쓰기하지 않는다.
- 사용자에게 `다른 기기에서 변경된 항목입니다.`를 표시한다.
- 선택:
  - `최신 내용 다시 불러오기`
  - `현재 입력으로 교체`
- 교체도 현재 사용자 소유권과 최신 버전을 다시 검증한다.

### 5.4 네트워크 오류

- 서버 저장에 실패하면 성공 상태를 표시하지 않는다.
- 원본 서버 데이터를 변경한 것으로 간주하지 않는다.
- 입력 폼을 유지하고 재시도를 제공한다.
- 오프라인 쓰기 큐는 만들지 않는다.

## 6. 공통 생성·수정·삭제 흐름

### 6.1 생성

1. 입력값 검증
2. 인증 사용자 확인
3. 새 ID 생성
4. `userId`를 인증 사용자 ID로 설정
5. 서버 저장
6. 서버 응답을 화면에 반영

### 6.2 수정

- `userId`, `id`, `createdAt`은 사용자가 변경할 수 없다.
- 저장 전 버전 충돌을 확인한다.
- 서버 성공 후 `updatedAt`을 갱신한다.

### 6.3 삭제

- 확인 대화상자 후 실행한다.
- RLS로 본인 행만 삭제할 수 있다.
- 관련 참조는 `DATA_MODEL.md`의 삭제 규칙에 따라 같은 사용자 범위에서 정리한다.
- 실패 시 기존 화면 데이터를 삭제된 것으로 표시하지 않는다.

## 7. 오늘 화면

모바일 순서:

1. 날짜·근무 상태
2. 오늘 꼭 끝낼 일
3. 회신 대기·확인 필요
4. 오늘 일정
5. 오늘 운동
6. 다음 개인 일정

PC 중앙 순서:

1. 오늘 날짜와 근무 상태
2. 이번 주 요약
3. 월간 통합 캘린더

PC 오른쪽 운영 열:

- 오늘 꼭 끝낼 일
- 회신 대기·확인 필요
- 회의·일정
- 오늘 운동
- 빠른 추가

표시 데이터는 현재 사용자 소유 데이터로 제한한다.

## 8. 일정

필수:

- 제목
- 영역
- 시작 날짜

선택:

- 종료 날짜
- 종일 여부
- 시작·종료 시간
- 메모
- 관련 링크
- 연결된 할 일
- 운동 일정 전용 정보

규칙:

- 종료 날짜는 시작 날짜보다 빠를 수 없다.
- 같은 날짜의 종료 시간은 시작 시간보다 늦어야 한다.
- 일정 하나에 현재 사용자의 여러 할 일을 연결할 수 있다.
- 할 일은 현재 사용자의 대표 일정 하나에만 연결할 수 있다.
- 일정 삭제 시 할 일은 유지하고 연결만 해제한다.
- 다른 사용자의 할 일 ID는 연결할 수 없다.

## 9. 할 일

필수:

- 제목
- 영역
- 상태
- 우선순위

선택:

- 수행일
- 마감일
- 후속 확인일
- 업무 단계
- 업무 대상
- 체크리스트
- 태그
- 색상
- 관련 링크
- 메모
- 대표 일정

상태:

- 예정
- 진행 중
- 회신 대기
- 확인 필요
- 완료
- 보류

우선순위:

- 낮음
- 보통
- 높음

기본 우선순위는 `보통`이다.

마감 표시:

- 마감일 이전 1~3일: 마감 임박
- 마감일이 오늘: 오늘 마감
- 마감일이 지남: 기한 경과
- 완료·보류: 마감 표시 제외

앱은 상태와 우선순위를 자동 변경하지 않는다.

## 10. 오늘 꼭 끝낼 일

- 날짜별 별도 컬렉션으로 관리한다.
- 현재 사용자의 할 일만 지정할 수 있다.
- 날짜별 최대 3개다.
- 완료된 항목은 당일 취소선과 함께 유지한다.
- 다음 날 자동 이월하지 않는다.
- 앱이 자동 선택·교체하지 않는다.

## 11. 회신 대기·확인 필요·후속 확인

- 회신 대기와 확인 필요는 저장 상태다.
- 후속 확인일은 마감일과 별도다.
- 오늘 또는 지난 후속 확인일의 미완료·비보류 할 일을 오늘 화면에 표시한다.
- 후속 확인일 도달만으로 상태를 변경하지 않는다.
- 같은 할 일이 여러 조건을 만족해도 같은 목록에서 한 번만 표시한다.

## 12. 캘린더

- 월간·주간 보기
- 월요일 시작
- 일정, 수행일이 있는 할 일과 마감일 표시
- 5개 영역 필터
- 현재 사용자 데이터만 표시
- 날짜 선택 후 빠른 추가
- 월간·주간 보기 모두 이전·다음 기간과 오늘 이동을 제공한다.
- 일정, 날짜가 있는 업무, 학교 날짜 스티커를 제목·설명·메모·장소로 검색하고 결과 날짜로 이동한다.
- 일정과 날짜가 있는 업무는 날짜 변경 메뉴를 기본으로 제공하고 PC에서는 같은 확인 흐름을 여는 드래그 이동도 지원한다.
- 반복 항목 이동은 `이 일정만`, `이 일정과 이후 일정`, `전체 반복 일정` 범위를 명시적으로 확인한다.
- 운동 스티커와 학교 날짜 스티커는 드래그 이동 대상에서 제외한다.
- 외부 캘린더 연동은 제외한다.

## 13. 반복 업무와 템플릿

### 13.1 반복 업무

- 할 일에서 매일·매주·매월·매년 중 하나를 사용자가 직접 지정한다.
- 수행일을 반복 기준일로 사용한다.
- 로그인 후 브리핑 또는 업무 화면을 열 때 기준일까지 누락된 업무를 자동 생성한다.
- 같은 반복 원본과 수행일 조합은 한 번만 생성한다.
- 월·연 반복에서 존재하지 않는 날짜는 건너뛴다.
- 생성된 업무는 현재 사용자 소유이며 기존 Task CRUD와 RLS를 그대로 적용한다.

### 13.2 반복 업무 템플릿

- 보건업무로 고정한다.
- 업무명, 체크리스트, 예상 시작 시점, 마감일 기준, 대상, 링크와 기본 메모를 저장한다.
- 현재 사용자 소유 템플릿만 조회·복사한다.
- 템플릿 자체는 자동 생성하지 않는다.
- 적용 연도와 날짜를 검토한 뒤 새 할 일로 복사한다.
- 템플릿 삭제가 기존 복사 업무를 삭제하지 않는다.

## 14. 운동

- 운동 일정과 운동 기록을 관리한다.
- 운동 일정: 제목, 날짜, 시간, 종류, 장소, 목표와 메모
- 운동 기록: 완료 여부, 실제 운동일, 강도, 컨디션과 메모
- 운동 일정 하나에 기록 하나만 연결한다.
- 운동 일정과 기록의 `userId`가 같아야 한다.
- 운동 일정 삭제 시 연결 기록도 삭제한다.
- 운동 완료가 할 일을 변경하지 않는다.

## 16. 빠른 메모

- 내용만 저장한다.
- 상태, 영역, 마감일과 완료 여부를 저장하지 않는다.
- 현재 사용자 소유 데이터다.
- 일정 또는 할 일로 전환할 수 있다.
- 전환 성공 후 원본 삭제 여부를 사용자가 선택한다.

## 17. 근무시간·교시 설정

- 사용자별 요일 근무 여부, 출퇴근 시간과 시간 구간을 저장한다.
- 같은 계정으로 로그인한 기기에서 같은 설정을 사용한다.
- 겹치는 시간 구간은 저장할 수 없다.
- 현재 교시와 퇴근까지 남은 시간은 계산하며 저장하지 않는다.

## 18. JSON 백업·복원

- 현재 로그인 사용자의 도메인 데이터만 내보낸다.
- 다른 사용자의 데이터와 인증 세션은 포함하지 않는다.
- 백업에 `ownerUserId`를 포함해 소유 사용자를 식별한다.
- 복원 시 로그인 사용자와 백업 소유자가 다르면 적용하지 않는다.
- 합치기는 같은 데이터 유형·동일 ID에서 최신 `updatedAt`을 유지한다.
- ID가 다르면 제목과 날짜가 같아도 별도 데이터다.
- 복원 행의 `userId`는 현재 인증 사용자로 검증하며 임의 변경을 허용하지 않는다.
- 전체 검증 성공 후 서버 트랜잭션 또는 실패 시 롤백 가능한 절차로 적용한다.

## 19. 개인정보 보호 UX

보건업무, 체크리스트, 빠른 메모, 템플릿과 백업 화면에 다음 안내를 표시한다.

> 학생 이름, 학번, 질병명, 상담 내용 등 개인 건강정보를 입력하지 마세요. 업무 상태와 비식별 수량만 기록해 주세요.

Google 로그인은 개인 데이터 동기화를 위한 것이며 학생정보 저장 허용을 의미하지 않는다.

## 20. 빈 상태

- 미로그인: `Google 계정으로 로그인해 BOGUNON을 시작하세요.`
- 오늘 꼭 끝낼 일 없음: `오늘 꼭 끝낼 일로 지정한 항목이 없습니다.`
- 일정 없음: `오늘 예정된 일정이 없습니다.`
- 회신 대기 없음: `회신을 기다리는 업무가 없습니다.`
- 운동 없음: `오늘 예정된 운동이 없습니다.`
- 동기화 데이터 없음: 첫 데이터 추가 행동을 제공한다.

## 21. 오류 상태

- 인증 실패: `Google 로그인을 완료하지 못했습니다. 다시 시도해 주세요.`
- 세션 만료: `로그인이 만료되었습니다. 다시 로그인해 주세요.`
- 조회 실패: `데이터를 불러오지 못했습니다. 다시 시도해 주세요.`
- 저장 실패: `저장하지 못했습니다. 입력한 내용은 유지됩니다.`
- 권한 오류: `이 데이터에 접근할 수 없습니다.`
- 충돌: `다른 기기에서 변경된 항목입니다.`
- 네트워크 오류: `인터넷 연결을 확인한 뒤 다시 시도해 주세요.`

오류 시 다른 사용자의 데이터, 행 ID, SQL과 내부 정책명을 표시하지 않는다.

## 23. 반응형 동작

모바일 하단 메뉴:

- 브리핑
- 업무함
- 일정
- 운동
- 설정

새 항목 생성은 모바일 하단 내비게이션 오른쪽 위의 플로팅 `+` 버튼에서 시작한다. 버튼은 업무, 일정, 업무 절차, 운동 기록, 빠른 메모와 작성 도움을 고르는 하단 시트를 열며, 선택 후 기존 생성·상세 흐름을 재사용한다. PC는 왼쪽 고정 사이드바, 중앙 운영 캔버스, 오른쪽 운영 열의 3열 브리핑을 사용한다. 오른쪽 운영 열은 오늘의 업무를 상시 요약하며, 항목 편집용 상세 패널은 별도로 기본 닫힘 상태를 유지한다. 모바일은 둥근 플로팅 하단 내비게이션과 단일 열 카드 흐름으로 전환한다. 인증 상태, 저장 중과 동기화 오류는 양쪽에서 같은 의미로 표시한다.

### 23.1 Phase 2.5 운영 대시보드 원칙

- PC 왼쪽 사이드바에는 브리핑, 업무, 업무 절차, 캘린더, 연간 계획, 운동과 설정을 배치한다.
- 중앙은 날짜·근무 상태, 주간 요약, 월간 통합 캘린더 순서로 구성한다.
- 오른쪽은 오늘의 업무, 회의·일정, 운동과 빠른 추가를 제공한다.
- 보건업무, 학교일정, 제출·마감, 회신 대기, 개인일정과 운동을 동일한 정보 체계에서 구분한다.
- 빠른 추가는 기존 생성 패널을 열며 Phase 3부터 업무와 일정을 Supabase에 저장한다.
- 학생 이름, 학번, 질병명, 상담 내용, 검사 결과를 화면 예시나 입력 모델에 포함하지 않는다.

## 28. Phase 5 보건업무 재사용과 연간 운영

- BOGUNON은 개인정보를 포함하지 않는 12개 기본 보건업무 템플릿을 애플리케이션 기본값으로 제공한다.
- 기본 템플릿은 불변이며 사용자는 자기 소유 템플릿으로 복제한 뒤 재사용한다.
- 템플릿 적용은 Task 또는 Event 생성 폼의 기본값만 채우며 사용자가 날짜와 내용을 검토하기 전에는 저장하지 않는다.
- Task 체크리스트는 순서와 완료 상태를 저장하고 `완료 수/전체 수`와 완료율을 표시한다. 미완료 항목이 있으면 Task 완료 전에 확인한다.
- 연간 업무 화면은 12개월의 Task와 Event, 마감·반복·완료 상태와 카테고리 색상을 제공한다.
- 완료 업무는 제목·카테고리·연도·기간·완료 상태로 찾고 새 업무 또는 사용자 템플릿으로 재사용한다.
- 한국어 빠른 입력은 날짜·시간·카테고리·우선순위·반복·종류를 규칙으로 추출하고 반드시 미리보기와 수정 후 저장한다.
- Task와 Event는 여러 HTTP·HTTPS 관련 링크와 알림 시점 데이터를 저장한다. 실제 알림 발송과 AI API는 Phase 5 범위가 아니다.
- 모든 템플릿·체크리스트·링크·알림 행은 인증 사용자 소유이며 RLS로 다른 사용자 접근을 차단한다.
- 자유 입력과 기본 예시는 개인을 식별하거나 건강 상태를 드러내는 정보를 저장하도록 유도하지 않는다.

## 29. Phase 6 Workflow OS

Phase 6는 Phase 5의 단일 Task·Event 템플릿과 복제 기능을 대체하지 않는다. 하나의 Task에 여러 단계로 이루어진 보건업무 절차를 연결하고, 현재 단계와 다음 행동을 일관된 상태 전이로 운영하는 Workflow OS를 추가한다.

### 29.1 템플릿과 인스턴스

- Workflow 템플릿은 이름, 설명, 카테고리, 기본 우선순위, 권장 시기, 순서가 있는 단계와 후속 업무 규칙을 정의한다.
- Phase 6 기본 템플릿은 `결핵검진`과 `학생건강검진`이며 불변이고 개인정보를 포함하지 않는다.
- 사용자는 기본 템플릿을 자기 소유 템플릿으로 복제하거나 직접 템플릿을 만들고 단계·체크리스트·링크·완료 조건·후속 규칙을 편집한다.
- 사용자가 미완료 Task에 템플릿을 적용하면 독립 실행 인스턴스를 만들고 템플릿 내용을 스냅샷으로 복사한다. 이후 템플릿 변경은 기존 인스턴스에 역동기화하지 않는다.
- 각 Workflow 인스턴스는 정확히 하나의 원본 Task에 연결한다. Phase 6 시작 화면은 이미 Workflow가 연결된 Task를 기본 후보에서 제외한다.
- Phase 5 `task_templates`는 한 개 Task·Event의 기본값 재사용을 담당하고, Workflow 템플릿은 Task 안의 여러 실행 단계를 담당한다.

### 29.2 상태와 전이

- Workflow 상태는 `active`, `paused`, `completed`, `cancelled`이다.
- 허용 전이는 `active → paused|completed|cancelled`, `paused → active|cancelled`이다. `completed`와 `cancelled`는 종료 상태이며 다시 열지 않는다.
- 단계 상태는 `pending`, `in_progress`, `completed`, `skipped`, `blocked`이다.
- 허용 단계 전이는 `pending → in_progress|skipped|blocked`, `in_progress → completed|skipped|blocked|pending`, `completed → in_progress`, `skipped → in_progress`, `blocked → in_progress|skipped|pending`이다.
- 단계 완료 시 미완료 체크리스트가 있으면 기본적으로 차단하고, 사용자가 명시적으로 강제 완료를 확인한 경우에만 허용한다.
- Workflow 완료는 모든 단계가 `completed` 또는 `skipped`일 때만 허용한다.
- 진행률은 전체 단계 수 대비 `completed` 단계 수로 계산하고, 남은 단계 수에서는 `completed`와 `skipped`를 제외한다.
- 다음 행동은 먼저 `in_progress`, 다음으로 `blocked`, 마지막으로 `pending` 단계를 선택한다. 현재 단계의 미완료 체크리스트를 일반 행동 문구보다 우선한다.

### 29.3 원자적 저장과 Task 연결

- 템플릿 묶음 저장, 인스턴스 시작, 단계 상세 저장, 단계 전이, Workflow 전이와 완료는 각각 서버 RPC의 단일 PostgreSQL 트랜잭션으로 처리한다.
- 인스턴스 시작은 인스턴스, 단계 스냅샷, 체크리스트, 링크, 후속 규칙과 타임라인을 함께 저장한다. 일부만 저장된 인스턴스를 허용하지 않는다.
- Workflow·단계 전이는 서버가 현재 상태, 허용 전이, 소유권과 완료 조건을 다시 검증한 뒤 상태·시각·현재 단계·타임라인을 함께 갱신한다.
- 단계 또는 Workflow 완료로 생성되는 후속 Task는 Phase 5 Task 필드·체크리스트·RLS를 그대로 사용한다. Task 생성이나 후속 규칙 갱신이 실패하면 전이 전체를 롤백한다.
- 인스턴스는 Task를 통해 업무 목록·검색·필터와 연결된다. Workflow 완료는 원본 Task를 같은 트랜잭션에서 완료하지만, Task만 먼저 완료해도 Workflow 단계 상태를 암묵적으로 변경하지 않는다.

### 29.4 화면 통합

- `/workflows`는 기본·사용자 템플릿 관리, Task에 Workflow 적용, 실행 인스턴스 검색·필터, 단계 편집과 타임라인을 제공한다.
- 브리핑 대시보드는 진행 중 Workflow 수, 보류 단계와 가장 가까운 다음 행동을 표시하고 `/workflows`의 해당 인스턴스로 연결한다.
- Workflow에서 생성된 후속 Task는 기존 업무 목록·브리핑·검색·필터에 일반 Task와 같은 규칙으로 나타난다.
- Desktop과 Mobile은 동일한 상태·진행률·다음 행동을 표시하며 작은 화면에서도 전이 행동을 숨기지 않는다.

### 29.5 개인정보 보호와 QA

- 템플릿, 예시, 필드명과 로그에는 학생 이름, 학번, 반·번호, 연락처, 질병명, 검진 결과, 상담 기록을 넣지 않는다.
- 대상은 학년·범주와 비식별 수량으로만 표현한다. 메모와 내부 참고사항에도 학생별 건강정보를 기록하지 않도록 안내한다.
- 모든 사용자 소유 Workflow 행과 연결 Task에는 RLS와 동일 사용자 관계 제약을 적용하고 비로그인 접근을 차단한다.
- QA는 상태 전이표, 진행률·다음 행동, RPC 실패 롤백과 동시 요청, 사용자 A/B 격리, 새로고침 유지, Task·대시보드 통합, Desktop·Mobile 반응형과 키보드 접근성을 포함한다.
- Phase 6 완료 전 Phase 5 템플릿·체크리스트·복제·연간 업무·빠른 입력·링크·알림 저장의 회귀 테스트를 통과해야 한다.

## 30. Phase 7 개인정보 안전 AI 업무 도우미

Phase 7은 기존 업무 흐름 위에 선택적인 AI 보조 계층을 추가한다. Phase 1~6의 기능과 데이터 계약을 대체하지 않으며 Phase 8은 시작하지 않는다.

### 30.1 AI 사용 목적과 진입점

- AI는 한국어 업무 입력 정리, Task·Event 초안, 우선순위·체크리스트·다음 행동 제안과 선택 항목 요약을 제공한다.
- 사용자는 PC 사이드바의 `AI 업무 도우미`와 브리핑, 빠른 추가, 업무, Workflow, 연간 업무의 문맥 버튼에서 공통 반응형 AI 패널을 연다.
- 문맥 진입은 현재 선택한 항목만 포함하고, 전역 진입은 사용자가 직접 붙여 넣거나 선택한 내용만 사용한다.
- AI를 사용하지 않아도 기존 생성·수정·Workflow 전이와 캘린더 기능을 모두 사용할 수 있다.

### 30.2 구조화된 미리보기와 확인

- AI 응답은 읽기 전용 답변·제안 또는 런타임 검증된 `mutation_preview`로 반환한다.
- 변경안은 대상, 현재 값과 제안 값을 구분해 표시하고 사용자가 항목별 포함 여부와 값을 수정할 수 있어야 한다.
- `확인하고 반영` 전에는 DB를 쓰지 않는다. 확인 후에도 기존 server action, Repository/RPC, 도메인 검증과 RLS를 사용한다.
- AI operation에는 삭제가 없다. 삭제·취소·일괄 제거와 Workflow 상태 전이는 AI가 실행하지 않으며 기존 제품 화면에서 사용자가 직접 수행한다.
- schema 오류, 취소, timeout, rate limit과 provider 실패에서는 쓰기가 0건이어야 한다. 저장 실패를 부분 성공처럼 표시하지 않는다.

### 30.3 Provider와 fallback

- UI는 provider에 종속되지 않으며 서버 provider interface만 호출한다.
- `AI_PROVIDER`, `OPENAI_API_KEY`, `AI_MODEL`은 서버 전용이다. 브라우저가 provider, 모델이나 자격 증명을 선택·조회할 수 없다.
- OpenAI provider와 결정적인 mock provider가 같은 입력·출력 계약을 사용한다.
- 개발 환경, 자격 증명 누락 또는 장애 시 mock을 사용할 수 있으나 응답에 mock임을 표시하며 실제 provider 응답으로 가장하지 않는다.
- 인증 사용자 기준 분당 10회 제한, 전체 12초 timeout을 적용하고 자동 재시도는 하지 않는다.

### 30.4 개인정보와 기록

- provider에는 요청에 필요한 최소 문맥만 전송한다. 토큰, 이메일, `user_id`, 내부 ID, 전체 목록과 다른 사용자 데이터는 제외한다.
- 메모·설명은 기본 제외하며 사용자가 명시적으로 AI 입력에 포함한 경우에만 전송한다.
- 학생 개인정보·건강정보 입력 금지 안내를 입력부와 확인 화면에 제공하고 입력·응답 원문을 로그에 남기지 않는다.
- AI 대화·제안 기록은 기본 off다. 기본 모드는 DB와 localStorage에 요청·응답을 보존하지 않는다.
- `ai_preferences.history_enabled`의 기본값은 `false`다. 기록은 사용자 설정과 해당 요청의 명시적 opt-in이 모두 있을 때만 `ai_requests`·`ai_action_drafts`에 저장하고 사용자별 RLS를 적용한다.

### 30.5 인수 조건

- mock과 OpenAI provider가 동일한 구조 검증과 no-delete 필터를 통과한다.
- Desktop·Tablet·Mobile에서 진입, 입력, loading, preview, confirm, success·error 흐름이 동일하다.
- 사용자 A/B와 비로그인 검증에서 AI 확인 이후의 쓰기도 기존 데이터 격리를 유지한다.
- Phase 1~6의 인증, Task·Event, 캘린더, 템플릿, 연간 업무와 Workflow 회귀 테스트가 통과한다.
# 모바일 일정·설정·운동 스티커 보완 계약 (2026-07-18)

- 모바일 브리핑은 날짜 다음에 가장 가까운 오늘 일정, 오늘 업무, 오늘 일정을 배치하고 주간 날짜·요약·오늘의 운동·월간 달력 순으로 이어진다.
- 운동 영역의 기본 생성은 Event 폼이 아니라 날짜와 운동 스티커를 선택하는 즉시 저장 흐름이다. 운동 시간과 메모는 저장된 기록에서만 선택적으로 추가한다.
- 기본 운동 스티커는 로컬 SVG와 안정적인 `icon_key`를 사용한다. 운영체제 emoji, 외부 이미지 URL과 원격 폰트를 사용하지 않는다.
- 설정 행이 없는 로그인 사용자는 연결 오류가 아니라 기본값이 채워진 설정 폼을 본다. 저장 시 사용자별 행을 upsert한다.
- 기존 Event 기반 운동 기록은 자동 삭제·변환하지 않고 레거시 일정으로 분리해 보존한다.

## 연간 플래너 실사용 보완 (Phase 8 아님)

- `/annual`은 1월부터 12월까지의 보건업무 제안을 보여 주는 일정·업무 생성 진입점이다. 제안은 법적·행정적 의무 목록으로 표현하지 않는다.
- 제안을 선택하면 기존 Task 또는 Event 생성 폼을 열어 제목, 설명, 예상 시간, 반복, 체크리스트와 카테고리를 채운다. 선택만으로 DB에 저장하지 않는다.
- 정확한 날짜가 없는 제안은 해당 연도·월만 안내하고 날짜를 비워 둔다. 사용자가 날짜를 확인하기 전에는 저장할 수 없다.
- 동일 제목의 기존 Task·Event를 조회해 추가됨·예정됨·완료됨 상태를 표시하며 별도 진행 상태를 저장하지 않는다.
- 기본 제안은 정적 프리셋이고 사용자 항목만 사용자별 RLS 테이블에 저장한다. 학생 개인정보와 개별 건강정보를 입력하도록 유도하지 않는다.
- `보건실 월간·연간 통계`는 사용자가 직접 수행할 일반 Task 제안명일 뿐이며, 통계 집계·분석 화면이나 새 데이터 기능을 추가하지 않는다.

## 빠른 보건업무 생성 동선 (Phase 8 아님)

- 업무 화면과 모바일 새로 만들기 메뉴에서 공통 보건업무 프리셋을 빠르게 선택할 수 있다.
- 연간 플래너와 동일한 정적 preset registry를 사용하며 별도 목록이나 DB 레코드를 중복 정의하지 않는다.
- 프리셋 선택은 기존 Task·Event 생성 폼에 제목, 설명, 예상 시간, 반복, 체크리스트, 카테고리와 알림을 채운다. 사용자가 날짜와 내용을 확인해 저장하기 전에는 DB를 변경하지 않는다.
- 최근 사용은 브라우저에서 최대 4개만 저장하며 사용자 업무 원문이나 학생 개인정보를 기록하지 않는다.
- 사용자는 공통 프리셋을 수정하지 않고 즐겨찾기, 사용자 지정 순서와 숨김 상태만 계정별로 저장한다. 즐겨찾기는 사용자 지정이고 최근 사용은 브라우저에서 자동 계산되는 별도 정보다.
- `기본 순서로 복원`은 즐겨찾기·순서·숨김만 초기화하며 최근 사용과 이미 생성한 Task·Event는 변경하지 않는다.
- 저장된 결과는 일반 Task·Event이므로 오늘 화면, 캘린더, 검색, 필터, 반복과 체크리스트 흐름을 그대로 따른다.
- 프리셋 정의와 생성 동선에는 별도 업무 테이블이 없고, 사용자별 표시 설정만 최소 RLS 테이블에 저장한다. 통계 또는 AI 추천 기능은 추가하지 않는다.

## 설치형 PWA 계약 (Phase 8 아님)

- 설치 이름, 화면 워드마크, 아이콘 대체 텍스트와 metadata의 제품명은 `BOGUNON`으로 통일한다.
- standalone 첫 실행에서는 한 번만 간단한 환영 안내를 표시하며 확인 여부는 로컬 기기에만 저장한다.
- 작은 favicon은 장식을 줄인 B 심볼을 사용하고, maskable 아이콘의 핵심 심볼은 safe zone 안에 둔다.
- manifest는 `/briefing`을 시작 경로로 사용하고 `standalone`, `portrait-primary`, `ko-KR`을 선언한다. 미로그인 사용자는 기존 보호 경로 인증으로 `/login`에 이동한다.
- Android는 `beforeinstallprompt`를 사용자 동작 뒤 실행하며 iOS는 Safari의 공유 메뉴와 홈 화면 추가 절차를 안내한다. standalone 실행 중에는 설치 안내를 숨긴다.
- Service Worker는 같은 origin의 명시된 정적 자산만 캐시한다. 인증·API 응답, 사용자 데이터, 세션·토큰과 `POST`·`PATCH`·`DELETE` 요청은 Cache Storage에 저장하지 않는다.
- 오프라인에서는 연결 안내만 제공한다. 완전한 오프라인 편집, 백그라운드 동기화와 Web Push는 범위에 포함하지 않는다.
